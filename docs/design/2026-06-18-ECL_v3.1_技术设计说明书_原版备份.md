## 1. 文档概述

### 1.1 背景

本文档为北京银行 ECL 减值计量系统 v3.1 的技术设计说明书，依据《ECL_v3.1_业务需求说明书》编制。系统遵循 IFRS 9 / CAS 22 准则，实现全行信用风险资产的预期信用损失减值计量。

### 1.2 范围

本文档覆盖五层架构（数据层 → 参数层 → 引擎层 → 计算层 → 报表层）的全量技术设计，包括：

- 系统架构与技术选型
- 各模块的包结构、类设计、核心流程
- 7+1 引擎的详细设计与核心伪码
- 数据库表设计
- REST API 接口定义
- 非功能需求设计（性能/安全/可扩展）

### 1.3 术语对照

| BRD 术语 | 技术设计对应 |
|----------|------------|
| 借据（asset） | `Asset` / `LoanFacility` |
| 客户 | `Customer` |
| 押品池 | `CollateralPool` |
| 风险分组 | `RiskGroup` |
| 方案（scheme） | 参数版本方案 |

## 5. 参数层详细设计

### 5.1 参数层总体设计

参数层采用 **"主数据 + 方案 + 明细"三表模式**，6 大模块统一架构。

#### 5.1.1 核心接口

```java
public interface ParameterService<T extends AbstractScheme> {
    /** 获取当前生效方案 */
    T getEffectiveScheme(String schemeType);
    /** 创建方案草稿 */
    String createDraft(T scheme);
    /** 发布方案 */
    void publishScheme(String schemeId);
    /** 方案对比 */
    SchemeDiff compareScheme(String schemeId1, String schemeId2);
}
```

#### 5.1.2 方案状态机

```java
public enum SchemeStatus {
    DRAFT("草稿"),
    PUBLISHED("已发布"),
    EFFECTIVE("已生效"),
    HISTORICAL("历史版本");
    
    public boolean canTransitionTo(SchemeStatus target) {
        return switch (this) {
            case DRAFT -> target == PUBLISHED;
            case PUBLISHED -> target == EFFECTIVE;
            case EFFECTIVE -> target == HISTORICAL;
            case HISTORICAL -> false;
        };
    }
}
```

#### 5.1.3 自动生效批处理（每日 01:00）

```
扫描所有 PUBLISHED 且 effective_date <= TODAY 的方案 → EFFECTIVE
扫描被新方案替代的旧 EFFECTIVE 方案 → HISTORICAL
```

### 5.2 风险分组参数（5.2）

#### 分组匹配算法（核心逻辑）

```java
public String matchGroup(AssetInput asset, String schemeId) {
    List<RiskGroupDetail> details = loadDetails(schemeId);
    
    for (RiskGroupDetail d : details) {
        if (matches(asset, d)) return d.getGroupId();
    }
    return "GRP_DEFAULT";  // 兜底
}

private boolean matches(AssetInput asset, RiskGroupDetail d) {
    return matchField(asset.getBusinessLine(), d.getBusinessLine())
        && matchField(asset.getCustomerType(), d.getCustomerType())
        && matchField(asset.getProductType(), d.getProductType())
        && matchField(asset.getIndustryCode(), d.getIndustryCode())
        && matchField(asset.getRegionCode(), d.getRegionCode())
        && matchField(asset.getCollateralType(), d.getCollateralType());
}

// NULL = 通配
private boolean matchField(String assetValue, String ruleValue) {
    return ruleValue == null || ruleValue.isEmpty() || ruleValue.equals(assetValue);
}
```

### 5.3 阶段划分参数（5.3）

核心枚举：

```java
public enum Stage {

// 9 种触发类型：逾期天数 / 评级下降 / 五级分类 / 违约标识 / 舆情等
public class StageResult {

```

### 5.4 PD 参数（5.4）

三情景 PD 模型：

```java
// PD_details = [{情景, 权重, PD值, PD类型}]
// 权重：乐观 20%, 基准 60%, 悲观 20%
// PD类型：12M / LIFETIME / 100% (Stage 3)
```

### 5.5 LGD 参数（5.5）

```java
// LGD = LGD_covered × 押品覆盖率 + LGD_uncovered × (1 - 押品覆盖率)
// 押品覆盖率 = min(押品价值 × 折扣率 / EAD, 100%)
```

### 5.6 CCF 参数（5.6）

按"承诺类型 + 期限范围"查表匹配 CCF 系数。

### 5.7 管理层叠加（5.7）

```

多条规则命中时：只选等效调整比例最高的 1 条，非累加
```

## 6. 引擎层详细设计

### 6.0 引擎调度器（核心编排）

#### 6.0.1 执行顺序

| Order | 引擎 | 并行 |
|-------|------|------|
| 0 | 6.1 风险分组 + 6.2 阶段划分 | ✅ 可并行 |
| 1 | 6.3 PD + 6.4 EAD | ✅ 可并行 |
| 2 | 6.5 LGD | ❌ 串行 |
| 3 | 6.6 ECL | ❌ 串行 |
| 4 | 6.7 管理层叠加 | ❌ 串行 |
| 5 | 6.8 最终输出 | ❌ 串行 |

#### 6.0.2 引擎接口

```java
public interface EclEngine {
    EngineType getType();
    void execute(JobContext ctx);
}

public class JobContext {
    private String jobId;
    private CalculateRequest request;
    private Map<String, Object> sharedData;  // 引擎间共享数据
}
```

#### 6.0.3 调度伪码

```
1. 按 Order 分组（Group 0~5）
2. 每组的引擎：
   - 1 个引擎 → 顺序执行
   - 2 个引擎 → CompletableFuture.allOf 并行执行
3. 引擎间通过 JobContext.sharedData 传递中间结果
```

### 6.1 风险分组引擎

**输入**：借据的 6 维属性（业务条线/客户类型/产品类型/行业/地区/担保）
**输出**：group_id（含 GRP_DEFAULT 兜底）
**逻辑**：遍历明细表，空值通配匹配，priority 排序取优先级最高

### 6.2 阶段划分引擎

**输入**：逾期天数/评级/五级分类/违约标识/舆情
**输出**：stage_result + trigger_type + exception_flag
**逻辑**：按阶段方案规则判断 Stage 1/2/3

### 6.3 PD 计算引擎

**输入**：group_id + rating_code + stage_result
**输出**：pd_details（三情景 PD JSON 数组）
**逻辑**：查 tbl_pd_curve 获取三情景 PD 值
**异常**：查不到 → 阻断（PD 无兜底）

### 6.4 EAD 计算引擎

**输入**：已提取余额 + 未使用额度 × CCF
**输出**：ead_total（含 on_bs / off_bs_allocated）
**核心公式**：EAD = drawn + undrawn × CCF
**异常**：EAD < 0 → 强制为 0

### 6.5 LGD 计算引擎

**输入**：押品池 + 池内 EAD 汇总
**输出**：lgd_value（押品池级，池内借据共享）
**有押品**：LGD_covered × 覆盖率 + LGD_uncovered × (1 - 覆盖率)
**无押品**：退化 LGD_uncovered 路径
**异常**：缺 discount_rate → 阻断

### 6.6 ECL 计算引擎

**输入**：ead_total + lgd_value + pd_details
**输出**：ecl_weighted + ecl_details（多情景 ECL JSON）
**公式**：

```
Stage 1: ECL = EAD × PD_12M × LGD
Stage 2: ECL = EAD × PD_LIFETIME × LGD  
Stage 3: ECL = EAD × 100% × LGD
加权: Σ(ECL_i × weight_i)
```

### 6.7 管理层叠加引擎

**输入**：ecl_weighted + ead_total + overlay 规则
**输出**：ecl_overlay_total + ecl_final + selected_overlay_id + effective_ratio
**核心规则**：多条命中时，只选等效调整比例最高的 1 条

### 6.8 最终输出引擎

**输入**：6.7 输出结果
**输出**：ecl_final（含审计字段）
**逻辑**：纯透传 + 对账校验
**对账**：abs(ecl_final_68 - ecl_final_67) ≥ 0.01 → 阻断 + 告警

## 10. 非功能设计

### 10.1 性能

| 指标 | 目标 | 措施 |
|------|------|------|
| 批量计算 | 100万笔 ≤ 2 小时 | 并行引擎 + 分批提交 |

| 并发 | 5 任务同时 | 线程池隔离 + 防重锁 |

## 附录：关键业务规则速查

| 规则 | 逻辑 |
|------|------|
| 空值通配 | 明细字段 NULL → 通用匹配 |
