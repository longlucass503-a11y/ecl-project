## 1. 文档概述

### 1.1 背景

本文档为北京银行 ECL 减值计量系统 v3.1 的技术设计说明书，依据《ECL_v3.1_业务需求说明书》编制。系统遵循 IFRS 9 / CAS 22 准则，实现全行信用风险资产的预期信用损失减值计量。

### 1.2 范围

本文档覆盖五层架构（数据层 → 参数层 → 引擎层 → 计算层 → 报表层）的全量技术设计，包括：

- 系统架构与技术选型
- 各模块的包结构、类设计、核心流程
- 7+1 引擎的详细设计与核心伪码
- 数据库表设计（以 ECL 减值方案为主线的统一版本控制）
- REST API 接口定义
- 非功能需求设计（性能/安全/可扩展）

### 1.3 术语对照

| BRD 术语 | 技术设计对应 |
|----------|------------|
| 借据（asset） | `Asset` / `LoanFacility` |
| 客户 | `Customer` |
| 押品池 | `CollateralPool` |
| 风险分组 | `RiskGroup` |
| 方案（scheme） | 顶层 `EclScheme`，全局版本控制单元 |
| ECL 减值方案 | `tbl_ecl_scheme`，所有参数模块的版本根 |
| 参数层 | `parameter` 模块 |
| 引擎层 | `engine` 模块 |

### 1.4 ID 生成规则

- **主键 ID**：统一使用 UUID，保证全局唯一
- **业务编码**：每张业务表额外提供人工可读编码字段（如 `scheme_code` / `group_code`），用于页面展示和模型人员识别
- **编码示例**：`SCH_001`、`GRP_001`、`JOB_20260618_001`
- **编码规则**：ID 不对外展示，前端和报表统一展示业务编码

### 2.3 触发流程（API 模式）

```
┌──────────┐     POST /api/v1/ecl/calculate
│ 上游系统   │ ──────────────────────────────────► ┌──────────┐
│ 信贷系统   │                                      │ ECL 系统  │
│ 核心系统   │ ◄────────────────────────────────── │          │
└──────────┘     { jobId, status }                └──────────┘

ECL 系统内部流程：
1. 接收请求 → 创建计算任务 (job_id)
2. 确定使用的 ECL 减值方案：
   a. 正式跑批：自动获取当前状态为 EFFECTIVE 的唯一方案
   b. 试算模式：从请求参数中获取指定的方案 ID
3. 异步执行引擎链（按选定方案的参数版本）：
   a. 参数加载 (按 scheme_id 从各参数表加载)
   b. 6.1 + 6.2 并行执行
   c. 6.3 + 6.4 并行执行
   d. 6.5 执行
   e. 6.6 执行
   f. 6.7 执行
   g. 6.8 输出 + 对账
4. 写报表表
5. 回调/轮询获取结果
```

## 5. 参数层详细设计

### 5.1 参数层总体设计

#### 5.1.1 设计哲学：以 ECL 减值方案为顶层管理单元

参数层采用 **"ECL 减值方案统一管理 + 各模块明细按 scheme_id 隔离"** 的架构。

核心变化（对比原 BRD）：

| 原设计 | 新设计 | 原因 |
|--------|--------|------|
| 各模块各自维护方案表（6 套方案/状态机） | 统一由 `tbl_ecl_scheme` 管理版本和状态 | 1 套状态机 vs 6 套，运维成本低 |
| 以 `tbl_risk_group` 作为全局主表关联 | 以 `tbl_ecl_scheme` 为顶层，各模块挂载其下 | CCF 等不依赖 group_id 的表挂靠自然 |
| 各模块方案独立发布 | ECL 方案统一定版、发布、生效 | 模型人员操作的是"一套方案"而非散装参数 |

#### 5.1.2 纯方案级版本控制

所有参数变更统一通过 ECL 方案版本管理，**不再有行级状态和行级生效时间**。

**核心原则：**

```
变更的唯一路径：复制方案 → DRAFT 修改 → 整体发布

不存在"在已生效方案内单行修改"的路径
不存在行级 row_status / effective_date
所有明细数据跟随所属 ECL 方案的版本状态
```

#### 5.1.3 两种变更场景

##### 场景一：全新创建 ECL 方案（从零开始）

```
新建 ECL 方案 → scheme_status = DRAFT
  └─ 新增各参数层数据
       （自由修改）
  ↓ 发布
  ├─ [立即生效] → ECL 方案直接 → EFFECTIVE
  └─ [计划生效] → 设置 effective_date → PUBLISHED
                   ↓ 到达生效日期（批处理 01:00）
                  ECL 方案 → EFFECTIVE
```

##### 场景二：整体调整 ECL 方案（全部复制）

```
SCH_001（EFFECTIVE）
  │ [模型人员点击"新建方案"]
  ▼
复制 SCH_001 全部数据 → SCH_002（新 scheme_id，版本号+1）
  │ SCH_001 不受影响，仍在正式跑批中使用
  │ 方案状态 = DRAFT
  │ [模型人员在新方案上自由修改]
  ▼
SCH_002（DRAFT）
  │ 可增删改任意参数
  │ [发布]
  ├─ [立即生效] → SCH_002 + 所有数据直接 → EFFECTIVE
  │                 SCH_001 + 所有数据直接 → EXPIRED
  └─ [计划生效] → 设置 effective_date
                   SCH_002 → PUBLISHED
                   ↓ 到达生效日期
                   SCH_002 → EFFECTIVE
                   SCH_001 → EXPIRED
```

#### 5.1.4 方案状态机

```java
public enum SchemeStatus {
    DRAFT("草稿"),
    PUBLISHED("已发布"),
    EFFECTIVE("已生效"),
    EXPIRED("已失效");

    public boolean canTransitionTo(SchemeStatus target) {
        return switch (this) {
            case DRAFT -> target == PUBLISHED;
            case PUBLISHED -> target == EFFECTIVE;
            case EFFECTIVE -> target == EXPIRED;
            case EXPIRED -> false;
        };
    }
}
```

#### 5.1.5 自动生效批处理（每日 01:00）

仅作用于选择"计划生效"的方案，"立即生效"的方案已当场处理完毕。

```
1. 扫描所有 PUBLISHED 且 effective_date <= TODAY 的 ECL 方案
   → 方案状态改为 EFFECTIVE
2. 扫描被新方案替代的旧 EFFECTIVE 方案
   → 旧方案状态改为 EXPIRED
```

#### 5.1.6 复制机制

当模型人员创建新方案时，系统自动执行全量复制：

### 5.2 风险分组参数（5.2）

#### 5.2.1 设计思路

风险分组采用 **"主数据 + 明细"两表模式**（去掉了原方案的方案表，版本控制统一由 `tbl_ecl_scheme` 管理）。

- `tbl_risk_group`：分组主数据，带 `scheme_id`，不同 ECL 方案可以有不同数量和定义的分组
  - 纯元数据（名称、排序、描述），不参与版本控制
  - 删除分组只需在 DRAFT 方案内删除对应行，老方案数据不受影响
- `tbl_risk_group_detail`：4 维匹配规则明细，带 `scheme_id` + `group_id`
- 匹配规则变化不影响 `group_id`，因此 PD/LGD/Stage/Overlay 等引用 `group_id` 的表不受影响

#### 5.2.2 分组匹配算法（核心逻辑）

```java
public String matchGroup(AssetInput asset, String schemeId) {
    List<RiskGroupDetail> details = loadDetails(schemeId);
    
    for (RiskGroupDetail d : details) {
        if (matches(asset, d)) return d.getGroupId();
    }
    return "GRP_DEFAULT";  // 兜底
}

private boolean matches(AssetInput asset, RiskGroupDetail d) {
    return matchField(asset.getBusinessLine(), d.getBusinessLine())
        && matchField(asset.getProductType(), d.getProductType())
        && matchField(asset.getIndustryCode(), d.getIndustryCode())
        && matchField(asset.getCollateralType(), d.getCollateralType());
}

// 仅规则值 NULL / 空 = 通配；源数据为空且规则非空时不命中
private boolean matchField(String assetValue, String ruleValue) {
    if (ruleValue == null || ruleValue.isEmpty()) return true;
    if (assetValue == null || assetValue.isEmpty()) return false;
    return ruleValue.equals(assetValue);
}
```

#### 5.2.3 取值逻辑与规则

```
资产入参（business_line + product_type + industry_code + collateral_type）
    ↓
1. 获取当前生效的 ECL 方案 scheme_id（正式跑批）
   或使用请求中指定的 scheme_id（试算）
    ↓
2. 查 tbl_risk_group_detail → scheme_id + 4 维度匹配（参数规则 NULL=通配）
    → 按 priority ASC 排序
    → 命中第一条 → 得到 group_id
    ↓
3. 查 tbl_risk_group → scheme_id + group_id
    → JOIN 得到 group_name
    ↓
4. 若无匹配 → 记录异常，进入兜底分组（GRP_DEFAULT）
    → exception_flag = 'Y'
```

### 5.3 阶段划分参数（5.3）

#### 5.3.1 阶段状态机

阶段划分本质上是一个三态状态机，按 IFRS 9 / CAS 22 准则将资产划分为三个阶段：

```
            ┌──────────────────────────────────────┐
            │                                      │
            ▼                                      │
        ┌────────┐   信用风险显著增加   ┌────────┐  │
   ───▶ │Stage 1 │ ─────────────────▶ │Stage 2 │  │
        │ 正常类  │ ◀───────────────── │ 关注类  │  │
        └────────┘  风险消除+观察期满足 └────────┘  │
            ▲                                  │    │
            │ 禁止直接回跳                      │    │
            │ （须经 Stage 2 中转）             ▼    │
            │                              ┌────────┐
            │         6个月观察期+还本付息   │Stage 3 │ ◀── 违约触发
            │         ┌─────────────────── │ 损失类  │
            │         │                    └────────┘
            │         │                         │
            │         ▼                         │
            │      ┌────────┐                   │
            └───── │Stage 2 │ ◀─────────────────┘
        （不允许）   │ 关注类  │   风险解除
                   └────────┘
```

**关键约束：**
- ❌ Stage 3 → Stage 1 禁止
- ✅ Stage 3 → Stage 2 → Stage 1（须经观察期中转）
- ✅ Stage 2 → Stage 3（直接，无中间过渡）

#### 5.3.2 表设计

阶段划分采用 **两条规则表** 的设计：

- `tbl_stage_rule`：存储 FORWARD（向前判定）和 ROLLBACK（回跳校验）两类规则
  - 触发条件使用 **JSON 字段** 存储，不同风险分组可定义不同的条件结构
  - 不再使用散列的 trigger 字段（避免稀疏表）
- `tbl_crr_rating_drop_rule`：内部评级 / 外部评级下降阈值表，按评级体系和评级机构配置，供 FORWARD 和 ROLLBACK 规则复用

```sql
CREATE TABLE tbl_stage_rule (
    rule_id          BIGINT         AUTO_INCREMENT PRIMARY KEY,
    scheme_id        VARCHAR(32)    NOT NULL COMMENT '所属 ECL 方案',
    group_id         VARCHAR(32)    NOT NULL COMMENT '关联风险分组',
    rule_type        VARCHAR(16)    NOT NULL COMMENT 'FORWARD（向前判定）/ ROLLBACK（回跳校验）',
    stage_from       VARCHAR(8)     COMMENT '当前阶段（ROLLBACK 必填，FORWARD 可为空）',
    stage_to         VARCHAR(8)     NOT NULL COMMENT '目标阶段',
    priority         INT            NOT NULL COMMENT '匹配优先级（越小越高）',
    observation_days INT            COMMENT '观察期天数（仅 ROLLBACK 使用）',
    conditions       JSON           NOT NULL COMMENT '触发条件 JSON，结构按 group 类型不同',

    FOREIGN KEY (scheme_id, group_id) REFERENCES tbl_risk_group(scheme_id, group_id),
    UNIQUE KEY uk_stage_rule (scheme_id, group_id, rule_type, stage_from, stage_to, priority)
) COMMENT '阶段划分规则表';

CREATE TABLE tbl_crr_rating_drop_rule (
    drop_rule_id     BIGINT         AUTO_INCREMENT PRIMARY KEY,
    scheme_id        VARCHAR(32)    NOT NULL COMMENT '所属 ECL 方案',
    group_id         VARCHAR(32)    NOT NULL COMMENT '关联风险分组',
    rating_system    VARCHAR(16)    NOT NULL COMMENT '评级体系：INTERNAL / EXTERNAL',
    rating_agency    VARCHAR(64)    NOT NULL COMMENT '评级机构：内部评级填 INTERNAL_CRR，外部评级填 Moody/S&P/Fitch 等',
    current_rating   VARCHAR(16)    NOT NULL COMMENT '当前评级（CRR1~CRR8 或国际评级 Aaa/Baa2 等）',
    drop_threshold   INT            NOT NULL COMMENT '下降≥N 级触发',

    FOREIGN KEY (scheme_id, group_id) REFERENCES tbl_risk_group(scheme_id, group_id),
    UNIQUE KEY uk_drop_rule (scheme_id, group_id, rating_system, rating_agency, current_rating)
) COMMENT '内部/外部评级下降幅度规则表';
```

#### 5.3.4 取值逻辑（核心算法）

```
输入：
  group_id
  last_stage          ← 借据的当前阶段（上期计算结果；首跑无历史时默认为 Stage 1）
  overdue_days, crr_final, ext_rating_this_year, is_npl, five_category, media...（借据字段）
  rating_system, rating_agency ← 根据业务分组和评级来源确定
  scheme_id           ← 绑定的 ECL 方案

═══ Step 1 — 向前判定（FORWARD） ═══

1. 查询 tbl_stage_rule →
   WHERE scheme_id = :scheme_id
     AND group_id = :group_id
     AND rule_type = 'FORWARD'
   ORDER BY priority ASC

2. 遍历规则，逐条解析 conditions JSON 匹配：
   ├── 规则 A JSON 中的条件全部满足 → stage_to = 该规则的目标
   │     → 进入 Step 2（回跳校验）
   └── 所有规则都不匹配 → stage_to = STAGE_1
        exception_flag = 'Y'
        → 进入 Step 2

═══ Step 2 — 回跳校验（ROLLBACK） ═══

3. 判断是否需要回跳校验：
   如果 stage_to < last_stage（阶段改善，如 3→2 或 2→1 或 3→1）
      → 需校验
   否则（阶段不变或恶化）
      → 直接返回 stage_to

   特殊约束：Stage 3 → Stage 1 禁止直接回跳，必须保持 Stage 3 或先回跳至 Stage 2

4. 查询 ROLLBACK 规则：
   WHERE scheme_id = :scheme_id
     AND group_id = :group_id
     AND rule_type = 'ROLLBACK'
     AND stage_from = :last_stage
     AND stage_to = :stage_to

5. 匹配规则中的 conditions JSON：
   ├── 满足条件 → 返回 stage_to（允许回跳）
   └── 不满足条件 → 返回 last_stage（保持原阶段，禁止回跳）
        exception_flag = 'Y'

评级下降级数计算：
  - 内部评级：使用客户评级表 crr_final 与 crr_int_last_year，评级机构固定为 INTERNAL_CRR
  - 外部评级：使用客户评级表 ext_rating_this_year / ext_rating_last_year，并同时使用 ext_rating_co_this_year / ext_rating_co_last_year
  - 本期和上期评级机构一致时才计算同一机构下的评级下降级数
  - rating_drop_levels = rating_rank(current_rating, rating_system, rating_agency) - rating_rank(last_rating, rating_system, rating_agency)
  - 用 (scheme_id, group_id, rating_system, rating_agency, current_rating) 查询 tbl_crr_rating_drop_rule.drop_threshold
  - rating_drop_levels >= drop_threshold 时触发评级下降条件

观察期判断：
  - ROLLBACK 规则的 observation_days 为上迁观察期参数
  - normal_consecutive_days >= observation_days 时，视为观察期满足
```

#### 5.3.5 引擎层伪码

```java
public StageResult determineStage(AssetInput asset, String schemeId, Stage lastStage) {
    if (lastStage == null) {
        lastStage = Stage.STAGE_1;  // 首跑或无历史结果时默认 Stage 1
    }

    // Step 1 — 向前判定
    List<StageRule> forwardRules = stageRuleMapper.selectList(
        new LambdaQueryWrapper<StageRule>()
            .eq(StageRule::getSchemeId, schemeId)
            .eq(StageRule::getGroupId, asset.getGroupId())
            .eq(StageRule::getRuleType, "FORWARD")
            .orderByAsc(StageRule::getPriority)
    );

    Stage targetStage = Stage.STAGE_1;  // 兜底
    boolean exceptionFlag = true;
    String matchedTriggerType = null;

    for (StageRule rule : forwardRules) {
        if (matchConditions(rule.getConditions(), asset)) {
            targetStage = rule.getStageTo();
            matchedTriggerType = extractTriggerType(rule.getConditions());
            exceptionFlag = false;
            break;
        }
    }

    // Step 2 — 回跳校验
    if (lastStage == Stage.STAGE_3 && targetStage == Stage.STAGE_1) {
        return new StageResult(Stage.STAGE_3, "ROLLBACK_BLOCKED_STAGE3_TO_STAGE1", true);
    }

    if (targetStage.ordinal() < lastStage.ordinal()) {
        // 阶段改善，需要回跳校验
        List<StageRule> rollbackRules = stageRuleMapper.selectList(
            new LambdaQueryWrapper<StageRule>()
                .eq(StageRule::getSchemeId, schemeId)
                .eq(StageRule::getGroupId, asset.getGroupId())
                .eq(StageRule::getRuleType, "ROLLBACK")
                .eq(StageRule::getStageFrom, lastStage.name())
                .eq(StageRule::getStageTo, targetStage.name())
        );

        boolean rollbackAllowed = false;
        for (StageRule rule : rollbackRules) {
            if (matchConditions(rule.getConditions(), asset)) {
                rollbackAllowed = true;
                break;
            }
        }

        if (!rollbackAllowed) {
            // 禁止回跳，保持原阶段
            return new StageResult(lastStage, "ROLLBACK_BLOCKED", true);
        }
    }

    return new StageResult(targetStage, matchedTriggerType, exceptionFlag);
}

/** 解析 JSON conditions 并匹配资产字段 */
private boolean matchConditions(JSONObject conditions, AssetInput asset) {
    // 通用逻辑引擎（支持 overdue_days / five_category / rating_drop / media_sentiment / is_npl 等）
    return ConditionEvaluator.evaluate(conditions, asset);
}
```

### 5.4 PD 参数（5.4）

#### 5.4.1 设计

PD（违约概率）是 ECL 公式 `ECL = PD × LGD × EAD` 的核心参数。每笔借据根据其 `group_id` + `rating_system` + `rating_agency` + `rating_code` 从 PD 曲线表获取对应值。

**表设计：**

- `tbl_pd_scenario`：情景定义表（乐观/基准/悲观），`scenario_type` 为 **自由文本**，支持未来扩展压力情景等新情景
- `tbl_pd_curve`：PD 曲线表，按 `scheme_id + group_id + scenario_id + rating_system + rating_agency + rating_code` 组织 12M PD 值

**核心特点：**

| 特性 | 设计 |
|------|------|
| 全行方案数 | 1 个（由 tbl_ecl_scheme 统一管理） |
| 分组差异化 | 通过 `tbl_pd_curve.group_id` 隔离 |
| 情景扩展性 | scenario_type 自由文本，新增情景只需加行 |
| 权重约束 | 同一方案下所有情景 weight 之和 = 1.0，发布前校验 |
| 兜底 | **无兜底**，缺值即阻断 |

**初始数据：**

| scenario_type | scenario_name | weight |
|--------------|--------------|--------|
| OPTIMISTIC | 乐观情景 | 0.20 |
| BASELINE | 基准情景 | 0.60 |
| PESSIMISTIC | 悲观情景 | 0.20 |

#### 5.4.2 取值逻辑

```
输入：scheme_id + group_id + rating_system + rating_agency + rating_code + stage_result
    ↓
1. 查 tbl_pd_scenario → scheme_id
   → 获取所有情景的 scenario_id + weight（如 3 行）
    ↓
2. 若 stage_result = Stage3：
   → 直接为每个情景生成 pd_value = 1.0，不查询 tbl_pd_curve
    ↓
3. 若 stage_result = Stage1 / Stage2：
   查 tbl_pd_curve → scheme_id + group_id + rating_system + rating_agency + rating_code + 各 scenario_id
   → 返回该评级在每个情景下的 12M pd_value（多行）
    ↓
4. 按阶段逐情景转换：
   Stage1：pd_value_i = PD_12M_i
   Stage2：pd_value_i = 1 - (1 - PD_12M_i)^(remaining_months / 12)
   Stage3：pd_value_i = 1.0
    ↓
5. 输出多情景 pd_details，ECL 计算层再按情景权重加权汇总
    ↓
异常：
  - Stage1 / Stage2 任一情景 PD 缺失 → 整笔资产阻断（PD 无兜底）
  - maturity_date 为空或 maturity_date <= calc_date → 整笔资产阻断
```

#### 5.4.3 引擎层伪码

```java
public List<PdDetail> loadPd(String schemeId,
                             String groupId,
                             String ratingSystem,
                             String ratingAgency,
                             String ratingCode,
                             Stage stage,
                             LocalDate maturityDate,
                             LocalDate calcDate) {
    // Step 1: 查情景
    List<PdScenario> scenarios = pdScenarioMapper.selectList(
        new LambdaQueryWrapper<PdScenario>()
            .eq(PdScenario::getSchemeId, schemeId)
    );
    if (scenarios.isEmpty()) {
        throw new EclException("ECL_004", "方案无 PD 情景配置");
    }

    if (maturityDate == null || !maturityDate.isAfter(calcDate)) {
        throw new EclException("ECL_005", "贷款到期日为空或不晚于计量日");
    }

    // Stage 3：PD 固定 100%，不依赖 PD 曲线
    if (stage == Stage.STAGE_3) {
        return scenarios.stream()
            .map(s -> new PdDetail(s, 1.0))
            .toList();
    }

    List<PdDetail> details = new ArrayList<>();
    for (PdScenario s : scenarios) {
        PdCurve curve = pdCurveMapper.selectOne(
            new LambdaQueryWrapper<PdCurve>()
                .eq(PdCurve::getSchemeId, schemeId)
                .eq(PdCurve::getGroupId, groupId)
                .eq(PdCurve::getRatingSystem, ratingSystem)
                .eq(PdCurve::getRatingAgency, ratingAgency)
                .eq(PdCurve::getRatingCode, ratingCode)
                .eq(PdCurve::getScenarioId, s.getScenarioId())
        );
        if (curve == null) {
            throw new EclException("ECL_001",
                "PD查询无结果: group_id=" + groupId
                + ", rating_system=" + ratingSystem
                + ", rating_agency=" + ratingAgency
                + ", rating_code=" + ratingCode);
        }

        double pd = stage == Stage.STAGE_1
            ? curve.getPdValue()
            : convertLifetimePd(curve.getPdValue(), maturityDate, calcDate);
        details.add(new PdDetail(s, pd));
    }

    return details;
}
```

### 5.5 LGD 参数（5.5）

#### 5.5.1 设计

LGD（违约损失率）是 ECL 公式 `ECL = PD × LGD × EAD` 的第二个核心参数。定义为违约时损失金额与 EAD 的比例。

**表设计：**

| 表 | 作用 | 是否依赖 group_id |
|----|------|-----------------|
| `tbl_lgd_curve` | LGD 基准值表，按 `(group_id, collateral_type, product_type)` 组合细分 | ✅ |
| `tbl_lgd_collateral_discount` | 押品折扣率表，按押品类型 / 市场属性配置折扣率 | ❌ |
| `tbl_lgd_depreciation` | 押品升值 / 折旧率表，按押品类型配置价值调整率 | ❌ |

**关键特性：**
- LGD 查不到 lgd_base_value 时：依次查 NONE 路径 → 使用方案级 default_lgd
- 押品池可覆盖价值按押品评估价值、折扣率、升值 / 折旧率计算净值后汇总
- 与 PD 不同（PD 仍为阻断，无兜底）

**与 PD 的关键区别：**

| | PD | LGD |
|--|----|-----|
| 情景 | ✅ 输出多情景明细，ECL 层按权重加权 | ❌ 单一基准值 |
| 细分维度 | group_id + rating_system + rating_agency + rating_code | group_id + collateral_type + product_type |
| 兜底 | ❌ 无兜底，阻断 | ✅ 可配置缺省值（方案 default_lgd，NONE 路径仍优先） |

#### 5.5.2 核心公式

```
对押品池：
  a. 逐笔押品计算净值：
     collateral_net_value_i = appraisal_value_i × (1 + depreciation_rate_i) × (1 - discount_rate_i)
     discount_rate_i        = 查 tbl_lgd_collateral_discount
     depreciation_rate_i    = 查 tbl_lgd_depreciation（升值为正，折旧为负）
  b. 汇总押品池净值：
     collateral_net_value = Σ(collateral_net_value_i)
  c. 汇总池内 EAD：
     EAD_total = Σ(押品池下借据.total_ead)
     借据.total_ead = 借据.on_bs_ead + 借据.off_bs_ead
     total_ead 直接复用 6.4 EAD 引擎输出，6.5 不重复计算 CCF
  d. 计算覆盖与未覆盖 EAD：
     EAD_covered   = min(collateral_net_value, EAD_total)
     EAD_uncovered = EAD_total - EAD_covered

押品池汇总：
  LGD_pool       = (EAD_covered × LGD_covered + EAD_uncovered × LGD_uncovered) / EAD_total

  其中：
    LGD_uncovered = 查 tbl_lgd_curve → (scheme_id, group_id, collateral_type, product_type)
    LGD_covered   = LGD_floor（覆盖部分 LGD 下限）
```

#### 5.5.3 取值逻辑

```
输入：scheme_id + group_id + collateral_pool_id + product_type + facility_pools
    ↓
1. 加载押品池信息：
   - 池基本信息
   - 池内全部押品清单
    ↓
2. 逐笔押品计算净值：
   discount_rate     = 查 tbl_lgd_collateral_discount
   depreciation_rate = 查 tbl_lgd_depreciation
   collateral_net_value_i = appraisal_value_i × (1 + depreciation_rate_i) × (1 - discount_rate_i)
   collateral_net_value = Σ(collateral_net_value_i)
    ↓
3. 加载押品池下全部子额度 facility_pools，并通过 facility_cd 找到池内借据
    ↓
4. 汇总押品池内 EAD：
   EAD_total = Σ(押品池下借据.total_ead)
   其中，借据.total_ead = 6.4 输出的 on_bs_ead + off_bs_ead
   6.5 不重新计算未使用额度 × CCF
    ↓
5. 计算可覆盖 EAD：
   EAD_covered = min(collateral_net_value, EAD_total)
   EAD_uncovered = EAD_total - EAD_covered
    ↓
6. 查询 LGD_uncovered：
   查 tbl_lgd_curve → scheme_id + group_id + collateral_type + product_type
    ↓
7. 计算 LGD_covered：
   LGD_covered = LGD_floor
    ↓
8. LGD_pool = (EAD_covered × LGD_covered + EAD_uncovered × LGD_uncovered) / EAD_total
    ↓
9. 押品池内每条借据 / 业务：
   lgd_value = LGD_pool
    ↓
异常：LGD_uncovered 查不到 lgd_base_value → 先查 collateral_type='NONE' 路径 → 仍未找到 → 使用方案 default_lgd
```

### 5.6 CCF 参数（5.6）

#### 5.6.1 设计

CCF（信用转换系数）用于将表外未提取额度转换为表内 EAD。**不依赖 group_id**，因为 CCF 是产品/承诺的监管属性，与业务分组无关。

**公式：** `表外 EAD = 未使用额度 × CCF`

**表设计（1 张表）：**

| 表 | 唯一键 | 是否依赖 group_id |
|----|-------|-----------------|
| `tbl_ccf_curve` | `(scheme_id, product_type, commitment_type, commitment_days_min, commitment_days_max)` | ❌ |

**关键特性：**
- 无情景表（CCF 是监管标准值，加情景无业务意义）
- 兜底机制：查不到 ccf_value → 使用方案 default_ccf（可配置，默认 0=不额外增加表外 EAD）
- 与 5.4 PD（多情景明细）差异化设计

#### 5.6.2 取值逻辑

```
输入：scheme_id + product_type + commitment_type + commitment_days
    ↓
查 tbl_ccf_curve → scheme_id + product_type + commitment_type
    + commitment_days BETWEEN commitment_days_min AND commitment_days_max
    → 得到 ccf_value
    ↓
异常：查不到 ccf_value → 使用方案 default_ccf（非阻断，用户可配置）

期限口径：
  commitment_days = DATEDIFF(额度信息表.facility_maturity_date, 额度信息表.facility_start_date)
  使用授信额度原始承诺期限，不使用跑批日到到期日的剩余期限
  facility_start_date / facility_maturity_date 缺失或期限小于等于 0 时，无法命中期限区间，使用 default_ccf
```

### 5.7 管理层叠加（5.7）

#### 5.7.1 设计

管理层叠加是 ECL 计算引擎输出后，管理层根据业务判断对特定范围进行人工干预调整的参数。

**表设计（1 张表）：**

```sql
CREATE TABLE tbl_overlay_rule (
    rule_id            BIGINT         AUTO_INCREMENT PRIMARY KEY,
    scheme_id          VARCHAR(32)    NOT NULL COMMENT '所属 ECL 方案',
    group_id           VARCHAR(32)    NOT NULL COMMENT '关联风险分组',
    overlay_type       VARCHAR(16)    NOT NULL COMMENT '调整类型标签：SPECIAL / CALIBRATION / MODEL_ERROR（仅供分类）',
    adjustment_target  VARCHAR(16)    NOT NULL DEFAULT 'ECL_FINAL' COMMENT '调整目标（目前仅 ECL_FINAL，预留扩展）',
    adjustment_type    VARCHAR(16)    NOT NULL COMMENT 'ADDBP / PERCENTAGE / FIXED',
    adjustment_value   DECIMAL(12,4)  NOT NULL COMMENT '调整值',
    priority           INT            NOT NULL COMMENT '匹配优先级（越小越高）',
    conditions         JSON           NOT NULL COMMENT '匹配条件 JSON，支持按行业、客户、产品、逾期、分类等现有源字段匹配',
    effective_date     DATE           NOT NULL COMMENT '规则生效日期',
    expiry_date        DATE           COMMENT '规则失效日期',

    FOREIGN KEY (scheme_id, group_id) REFERENCES tbl_risk_group(scheme_id, group_id)
) COMMENT '管理层叠加规则表';
```

**核心规则：**
1. **基数固定为 EAD**：叠加金额 = EAD × 调整比例
2. **作用于 ECL_FINAL**：`ECL_final = ECL_calculated + 叠加金额`
3. **非累加**：多条规则命中时只选等效调整比例最高的 1 条；等效比例相同时取 priority 最小
4. **匹配条件用 JSON**，与 5.3 阶段划分风格一致

#### 5.7.2 conditions JSON 约定

```json
-- 按行业范围匹配（使用借据信息表 industry_cn）
{"industry_cn": {"in": ["房地产", "建筑业"]}}

-- 按产品类型匹配
{"product_type": {"in": ["公司贷款", "银团贷款"]}}

-- 按逾期天数范围匹配
{"overdue_days": {"min": 31}}

-- 按五级分类匹配
{"loan_classif_cd": {"in": ["关注", "次级", "可疑", "损失"]}}

-- GROUP 范围（直接命中 group_id，不写 conditions）
-- 当 apply_scope 为 GROUP 时，conditions 可为空
```

#### 5.7.3 调整方式

| adjustment_type | 含义 | 叠加金额公式 |
|---------------|------|------------|
| ADDBP | 加基点 | EAD × (adjustment_value / 10000) |
| PERCENTAGE | 加百分比 | EAD × adjustment_value |
| FIXED | 固定金额 | adjustment_value（不依赖 EAD） |

#### 5.7.4 取值逻辑

```
输入：scheme_id + group_id + EAD + ECL_calculated + 借据维度字段
    ↓
1. 查 tbl_overlay_rule → scheme_id + group_id + 生效规则
   WHERE effective_date <= calc_date
     AND (expiry_date IS NULL OR expiry_date >= calc_date)
   → 按 priority ASC 排序
    ↓
2. 逐条解析 conditions JSON 匹配
   → 收集所有命中的规则
    ↓
3. 计算各命中规则的等效调整比例：
   ADDBP：     等效比例 = adjustment_value / 10000
   PERCENTAGE：等效比例 = adjustment_value
   FIXED：     等效比例 = adjustment_value / EAD
    ↓
4. 选等效比例最高的 1 条（非累加）；等效比例相同时取 priority 最小
   → 按选中的 adjustment_type 公式计算叠加金额
    ↓
5. ECL_final = ECL_calculated + 叠加金额
```

#### 5.7.5 `overlay_type` 说明

| overlay_type | 含义 | 影响计算 |
|-------------|------|---------|
| SPECIAL | 专项调整（如房地产行业专项） | ❌ 仅标签 |
| CALIBRATION | 模型校准调整 | ❌ 仅标签 |
| MODEL_ERROR | 模型误差补偿 | ❌ 仅标签 |

`overlay_type` 仅为分类标签，供前端筛选和审计追溯使用，不参与引擎计算逻辑。

```java
public class RiskGroupingEngine implements EclEngine {
    
    @Override
    public void execute(JobContext ctx) {
        String schemeId = ctx.getSchemeId();
        this.rules = loadRules(schemeId);
        this.groupMap = loadGroupMap(schemeId);

        for (CustomerContext customer : ctx.getCustomers()) {
            for (AssetInput asset : customer.getAssets()) {
            String groupId = matchGroup(asset, rules, groupMap);
            asset.setGroupId(groupId);
            if ("GRP_DEFAULT".equals(groupId)) {
                asset.setGroupException("Y");
            }
        }
    }
    
    private String matchGroup(AssetInput asset) {
        for (RiskGroupDetail rule : rules) {
            // 4 维度 AND 匹配（仅规则值 NULL / 空 = 通配）
            // 使用 ConditionEvaluator 统一匹配
            if (matchRule(asset, rule)) {
                return rule.getGroupId();
            }
        }
        // 兜底
        return "GRP_DEFAULT";
    }
    
    private boolean matchRule(AssetInput asset, RiskGroupDetail rule) {
        return ConditionEvaluator.matchField(asset.getBusinessLine(), rule.getBusinessLine())
            && ConditionEvaluator.matchField(asset.getProductType(), rule.getProductType())
            && ConditionEvaluator.matchField(asset.getIndustryCode(), rule.getIndustryCode())
            && ConditionEvaluator.matchField(asset.getCollateralType(), rule.getCollateralType());
    }
}

// ConditionEvaluator —— 统一条件评估器，三个引擎共享
public class ConditionEvaluator {
    
    /** 字段匹配：仅规则值 NULL / 空 = 通配；源数据为空且规则非空时不命中 */
    public static boolean matchField(String assetValue, String ruleValue) {
        if (ruleValue == null || ruleValue.isEmpty()) return true;  // 规则通配
        if (assetValue == null || assetValue.isEmpty()) return false; // 源数据缺失，不命中非空规则
        return ruleValue.equals(assetValue);
    }
    
    /** JSON conditions 匹配（6.2、6.7 使用） */
    public static boolean evaluate(JSONObject conditions, Map<String, Object> input) {
        if (conditions == null || conditions.isEmpty()) return true;
        for (String key : conditions.keySet()) {
            Object condition = conditions.get(key);
            Object value = input.get(key);
            if (!evaluateField(condition, value)) return false;
        }
        return true;
    }
    
    private static boolean evaluateField(Object condition, Object value) {
        if (condition instanceof JSONObject) {
            JSONObject cond = (JSONObject) condition;
            if (cond.containsKey("min") || cond.containsKey("max")) {
                // 范围匹配（如 overdue_days: {min:31, max:90})
                return matchRange(cond, value);
            }
            if (cond.containsKey("in")) {
                // 枚举匹配（如 five_category: {in:["次级","可疑","损失"]})
                return matchIn(cond, value);
            }
        }
        // 精确匹配 / OR 逻辑
        return false;
    }
}
```

```java
public class StageEngine implements EclEngine {

    @Override
    public void execute(JobContext ctx) {
        String schemeId = ctx.getSchemeId();

        // 一次性加载规则和评级下降阈值，内存中按 group_id / 评级机构筛选
        List<StageRule> allForwardRules = loadRules(schemeId, "FORWARD");
        List<StageRule> allRollbackRules = loadRules(schemeId, "ROLLBACK");
        Map<RatingDropKey, Integer> ratingDropMap = loadRatingDropMap(schemeId);

        for (CustomerContext customer : ctx.getCustomers()) {
            for (AssetInput asset : customer.getAssets()) {
                List<StageRule> forwardRules = filterByGroup(allForwardRules, asset.getGroupId());
                List<StageRule> rollbackRules = filterByGroup(allRollbackRules, asset.getGroupId());
                StageResult result = determineStage(asset, forwardRules, rollbackRules, ratingDropMap);
                asset.setStageResult(result);
            }
        }
    }

    private StageResult determineStage(AssetInput asset,
                                        List<StageRule> forwardRules,
                                        List<StageRule> rollbackRules,
                                        Map<RatingDropKey, Integer> ratingDropMap) {

        Stage lastStage = asset.getLastStage() == null ? Stage.STAGE_1 : asset.getLastStage();

        // ═══ Step 1: FORWARD 向前判定 ═══
        // 顺序：先检视 Stage 3 → 再 Stage 2 → 最后 Stage 1（兜底）
        Stage targetStage = Stage.STAGE_1;
        String matchedTrigger = null;
        boolean exceptionFlag = true;

        for (StageRule rule : forwardRules) {
            if (ConditionEvaluator.evaluate(rule.getConditions(), asset.toFieldMap(), ratingDropMap, rule.getObservationDays())) {
                targetStage = rule.getStageTo();
                matchedTrigger = extractTriggerType(rule.getConditions());
                exceptionFlag = false;
                break;
            }
        }

        // ═══ Step 2: ROLLBACK 回跳校验 ═══
        if (lastStage == Stage.STAGE_3 && targetStage == Stage.STAGE_1) {
            return new StageResult(Stage.STAGE_3, "ROLLBACK_BLOCKED_STAGE3_TO_STAGE1", true);
        }

        if (targetStage.ordinal() < lastStage.ordinal()) {
            // 阶段改善，需要回跳校验
            boolean passRollback = false;
            for (StageRule rule : rollbackRules) {
                if (rule.getStageFrom() == lastStage
                        && rule.getStageTo() == targetStage) {
                    if (ConditionEvaluator.evaluate(rule.getConditions(), asset.toFieldMap(), ratingDropMap, rule.getObservationDays())) {
                        passRollback = true;
                        break;
                    }
                }
            }

            if (!passRollback) {
                // 禁止回跳，保持原阶段
                return new StageResult(lastStage, "ROLLBACK_BLOCKED", true);
            }
        }

        return new StageResult(targetStage, matchedTrigger, exceptionFlag);
    }
}

/** ConditionEvaluator 扩展——支持按评级体系和评级机构查评级下降阈值 */
public class ConditionEvaluator {

    /** 基础字段匹配（6.1 使用） */
    public static boolean matchField(String assetValue, String ruleValue) { ... }

    /** JSON conditions 匹配（6.2、6.7 使用） */
    public static boolean evaluate(JSONObject conditions, Map<String, Object> input,
                                    Map<RatingDropKey, Integer> ratingDropMap,
                                    Integer observationDays) {
        // ... 支持 overdue_days / five_category / rating_drop / media_sentiment / is_npl 等
        for (String key : conditions.keySet()) {
            Object cond = conditions.get(key);
            if (key.equals("or")) {
                // OR 逻辑：任一子条件满足即通过
                JSONArray orItems = (JSONArray) cond;
                boolean anyMatch = false;
                for (Object item : orItems) {
                    if (evaluateSingle((JSONObject) item, input, ratingDropMap, observationDays)) {
                        anyMatch = true;
                        break;
                    }
                }
                if (!anyMatch) return false;
            } else if (key.equals("and")) {
                JSONArray andItems = (JSONArray) cond;
                for (Object item : andItems) {
                    if (!evaluateSingle((JSONObject) item, input, ratingDropMap, observationDays)) return false;
                }
            } else if (key.equals("rating_drop") && ratingDropMap != null) {
                JSONObject rating = (JSONObject) cond;
                RatingContext ratingCtx = resolveRatingContext(rating, input);
                int dropLevels = calcRatingDropLevels(ratingCtx);
                int threshold = ratingDropMap.getOrDefault(ratingCtx.toDropKey(), Integer.MAX_VALUE);
                if (dropLevels < threshold) return false;
            } else if (key.equals("rating_recovery")) {
                JSONObject rating = (JSONObject) cond;
                RatingContext ratingCtx = resolveRatingContext(rating, input);
                int recoveryLevels = calcRatingRecoveryLevels(ratingCtx);
                int minRecovery = rating.getIntValue("min_recovery_levels");
                if (recoveryLevels < minRecovery) return false;
            } else {
                if (!evaluateSingle(conditions, input, ratingDropMap, observationDays)) return false;
            }
        }
        return true;
    }
}
```

```java
public class PdEngine implements EclEngine {

    // 缓存：Map<(groupId, ratingSystem, ratingAgency, ratingCode, scenarioId), pdValue>
    private Map<String, Double> pdCache;

    @Override
    public void execute(JobContext ctx) {
        String schemeId = ctx.getSchemeId();

        // 一次性加载 PD 情景列表
        List<PdScenario> scenarios = loadScenarios(schemeId);
        // 一次性加载 PD 曲线到内存 Map
        this.pdCache = loadPdCurveMap(schemeId);

        for (CustomerContext customer : ctx.getCustomers()) {
            for (AssetInput asset : customer.getAssets()) {
                RatingContext rating = resolveRatingContext(asset);
                List<PdDetail> details = new ArrayList<>();

                if (asset.getMaturityDate() == null
                        || !asset.getMaturityDate().isAfter(ctx.getCalcDate())) {
                    asset.setPdException("ECL_005");
                    continue;
                }

                if (asset.getStageResult() == Stage.STAGE_3) {
                    for (PdScenario s : scenarios) {
                        details.add(new PdDetail(s, rating, 1.0));
                    }
                    asset.setPdDetails(details);
                    continue;
                }

                for (PdScenario s : scenarios) {
                    Double pdValue = pdCache.get(key(asset.getGroupId(),
                            rating.getRatingSystem(),
                            rating.getRatingAgency(),
                            rating.getRatingCode(),
                            s.getScenarioId()));

                    if (pdValue == null) {
                        // PD 无兜底，任一情景缺失则整笔资产阻断
                        asset.setPdException("ECL_001");
                        details.clear();
                        break;
                    }

                    // 按 stage_result 转换 PD 值
                    double pd = convertPdByStage(pdValue, asset.getStageResult(),
                            asset.getMaturityDate(), ctx.getCalcDate());

                    details.add(new PdDetail(s, rating, pd));
                }

                if (asset.getPdException() == null) {
                    asset.setPdDetails(details);
                }
            }
        }
    }

    private double convertPdByStage(double pd12m, Stage stage, LocalDate maturity, LocalDate calcDate) {
        return switch (stage) {
            case STAGE_1 -> pd12m;                              // 12M PD，直接用
            case STAGE_2 -> {
                // 永续 PD：1 - (1 - PD_12M)^T
                double t = ChronoUnit.MONTHS.between(calcDate, maturity) / 12.0;
                if (t < 1) t = 1;                               // 不足 1 年按 1 年
                yield 1 - Math.pow(1 - pd12m, t);
            }
            case STAGE_3 -> 1.0;                                 // 全覆盖
        };
    }
}
```

**设计说明：**
- EAD 是整个引擎链中最复杂的环节，涉及还款计划折现、表外 CCF 分配、跨借据分摊
- 以客户为中心，一次性处理该客户下的所有借据和额度
- CCF 参数一次性加载到内存
- `due_interest` 是还款计划信息表字段，表示当期应还利息；EAD 折现时仅用于未来期次现金流
- 表外业务不区分已开展 / 未开展，统一按额度未使用金额计算：`off_bs_ead = undrawn_amount × CCF`
- 表外 EAD 先按 `facility_cd` 在额度层计算，再分摊到同一 `facility_cd` 下的借据；多笔借据按借据放款额度 `amt_financed_cny` 比例分摊
- 无还款计划时仍按当前既定口径计算：`outstanding_balance + accrued_interest`

**核心逻辑：**

```java
public class EadEngine implements EclEngine {

    private Map<String, List<CcfValue>> ccfCache; // (productType, commitmentType) → 期限区间 CCF 列表
    private double defaultCcf;                // 方案级 CCF 缺省值

    @Override
    public void execute(JobContext ctx) {
        String schemeId = ctx.getSchemeId();
        this.ccfCache = loadCcfMap(schemeId);
        this.defaultCcf = ctx.getDefaultCcf();

        for (CustomerContext customer : ctx.getCustomers()) {
            // Step 1: 算表内 EAD（逐笔借据）
            for (AssetInput asset : customer.getAssets()) {
                double onBsEad = calcOnBsEad(asset);
                asset.setOnBsEad(onBsEad);
            }

            // Step 2: 算表外 EAD（按 facility_cd 逐额度计算）
            List<OffBsSubItem> offBsItems = calcOffBsEad(customer);

            // Step 3: 表外 EAD 分配到借据
            allocateOffBsToAssets(offBsItems, customer.getAssets());

            // Step 4: 汇总每条借据的 total_ead
            for (AssetInput asset : customer.getAssets()) {
                double total = asset.getOnBsEad() + asset.getOffBsEadAllocated();
                asset.setTotalEad(total);
            }
        }
    }

    /** 表内 EAD */
    private double calcOnBsEad(AssetInput asset) {
        if (asset.hasRepaymentSchedule()) {
            // 有还款计划：仅折现未来现金流 + 逾期欠款 + 应计利息
            double pv = 0;
            for (RepaymentSchedule installment : asset.getSchedule()) {
                if (!installment.getDueDate().isAfter(asset.getCalcDate())) {
                    continue;
                }
                double t = ChronoUnit.DAYS.between(asset.getCalcDate(), installment.getDueDate()) / 365.0;
                pv += (installment.getPrincipal() + installment.getInterest()) / Math.pow(1 + ctx.getDiscountRate(), t);
            }
            return pv + asset.getOverduePrincipal() + asset.getOverdueInterest() + asset.getAccruedInterest();
        } else {
            // 无还款计划：剩余本金 + 应计利息
            return asset.getOutstandingBalance() + asset.getAccruedInterest();
        }
    }

    /** 表外 EAD：按 facility_cd 逐额度计算，统一采用未使用额度 × CCF */
    private List<OffBsSubItem> calcOffBsEad(CustomerContext customer) {
        List<OffBsSubItem> result = new ArrayList<>();

        List<FacilityInfo> offBalanceFacilities = customer.getFacilities().stream()
            .filter(f -> f.getType() == FacilityType.OFF_BALANCE_SHEET)
            .toList();

        for (FacilityInfo item : offBalanceFacilities) {
            double ccf = lookupCcf(item);
            double undrawn = resolveUndrawnAmount(item);
            double ead = Math.max(undrawn, 0) * ccf;
            result.add(new OffBsSubItem(item, ead));
        }
        return result;
    }

    /** 表外 EAD 分配到借据 */
    private void allocateOffBsToAssets(List<OffBsSubItem> offBsItems, List<AssetInput> assets) {
        for (OffBsSubItem item : offBsItems) {
            // 按 facility_cd 找到该额度下的候选借据
            List<AssetInput> candidates = assets.stream()
                .filter(a -> Objects.equals(a.getFacilityCd(), item.getFacilityCd()))
                .toList();

            if (candidates.isEmpty()) {
                item.setException("Y");  // 无匹配借据，标记异常
                continue;
            }

            if (candidates.size() == 1) {
                // 1 对 1：全额分配
                candidates.get(0).addOffBsEad(item.getEadValue());
            } else {
                // 1 对 N：按借据放款额度比例分配
                double totalFinanced = candidates.stream()
                    .mapToDouble(AssetInput::getAmtFinancedCny)
                    .sum();
                if (totalFinanced <= 0) {
                    item.setException("Y");
                    continue;
                }
                for (AssetInput asset : candidates) {
                    double ratio = asset.getAmtFinancedCny() / totalFinanced;
                    asset.addOffBsEad(item.getEadValue() * ratio);
                }
            }
        }
    }

    private double lookupCcf(FacilityInfo item) {
        int commitmentDays = calcCommitmentDays(item.getFacilityStartDate(), item.getFacilityMaturityDate());
        return ccfCache.getOrDefault(key(item.getProductType(), item.getCommitmentType()), List.of()).stream()
            .filter(c -> commitmentDays > 0
                      && commitmentDays >= c.getCommitmentDaysMin()
                      && commitmentDays <= c.getCommitmentDaysMax())
            .findFirst()
            .map(CcfValue::getCcfValue)
            .orElse(defaultCcf);
    }

    private int calcCommitmentDays(LocalDate startDate, LocalDate maturityDate) {
        if (startDate == null || maturityDate == null) return -1;
        return (int) ChronoUnit.DAYS.between(startDate, maturityDate);
    }

    private double resolveUndrawnAmount(FacilityInfo item) {
        if (item.getUndrawnAmtCny() != null) return item.getUndrawnAmtCny();
        return item.getLimitAmtCny() - item.getUsedLimit();
    }
}
```

**设计说明：**
- 以押品池为起点计算池级 LGD，再赋值给池内每笔借据
- 6.5 不重复计算 EAD，也不重复计算未使用额度 × CCF；池内 EAD 直接汇总 6.4 输出的资产级 `total_ead`
- 参数（`lgd_base_value`、押品折扣率、押品升值 / 折旧率）一次性加载到内存
- `collateral_net_value` 表示押品净值汇总，不是源表 `collateral_value` 直接汇总

**核心公式：**

```
LGD_pool = (EAD_uncovered × LGD_uncovered + EAD_covered × LGD_covered) / EAD_total

其中：
  EAD_total       = Σ(押品池下借据.total_ead)
  借据.total_ead  = 借据.on_bs_ead + 借据.off_bs_ead
  collateral_net_value = Σ(appraisal_value × (1 + depreciation_rate) × (1 - discount_rate))
  EAD_covered     = min(collateral_net_value, EAD_total)
  EAD_uncovered   = EAD_total - EAD_covered
  LGD_uncovered   = tbl_lgd_curve.lgd_base_value（按 scheme_id + group_id + collateral_type + product_type）
  LGD_covered     = LGD_floor
```

**核心逻辑：**

```java
public class LgdEngine implements EclEngine {

    // 参数缓存（一次性加载）
    private Map<String, Double> lgdCurveMap;     // (groupId, collType, prodType) → lgd_base_value
    private Map<String, Double> discountMap;     // collateralType / marketAttribute → discount_rate
    private Map<String, Double> depreciationMap; // collateralType → depreciation_rate
    private double defaultLgd;                   // 方案级 LGD 缺省值
    private double lgdFloor;                     // 覆盖部分 LGD 下限

    @Override
    public void execute(JobContext ctx) {
        String schemeId = ctx.getSchemeId();
        loadParameterCache(schemeId);
        this.defaultLgd = ctx.getDefaultLgd();

        for (CustomerContext customer : ctx.getCustomers()) {
            // 按押品池分组处理
            Map<String, List<CollateralInfo>> poolMap = groupByCollateralPool(customer.getCollaterals());

            for (Map.Entry<String, List<CollateralInfo>> entry : poolMap.entrySet()) {
                String poolId = entry.getKey();
                List<CollateralInfo> collaterals = entry.getValue();

                // 找到该押品池关联的子额度和借据
                List<FacilityInfo> poolFacilities = findFacilitiesByPool(customer.getFacilities(), poolId);
                List<AssetInput> poolAssets = findAssetsByFacilities(customer.getAssets(), poolFacilities);

                // Step 1: 逐笔押品计算净值并汇总
                double collateralNetValue = 0;
                for (CollateralInfo coll : collaterals) {
                    double discountRate = lookupDiscountRate(coll);
                    double depreciationRate = lookupDepreciationRate(coll);
                    double netValue = coll.getAppraisalValue()
                        * (1 + depreciationRate)
                        * (1 - discountRate);
                    collateralNetValue += netValue;
                }

                // Step 2: 汇总押品池下所有借据 EAD，直接复用 6.4 total_ead
                double eadTotal = poolAssets.stream()
                    .mapToDouble(AssetInput::getTotalEad)
                    .sum();

                if (eadTotal <= 0) {
                    continue;
                }

                // Step 3: 计算可覆盖 / 未覆盖 EAD
                double eadCovered = Math.min(collateralNetValue, eadTotal);
                double eadUncovered = eadTotal - eadCovered;

                // Step 4: LGD_uncovered：按 group_id + collateral_type + product_type 查询
                String groupId = resolvePoolGroupId(poolAssets);
                String primCollType = collaterals.get(0).getCollateralType();
                String primProdType = resolvePoolProductType(poolAssets);
                double lgdUncovered = lgdCurveMap.getOrDefault(
                    key(groupId, primCollType, primProdType), defaultLgd);

                // Step 5: LGD_covered：覆盖部分 LGD 下限
                double lgdCovered = lgdFloor;

                // Step 6: 计算池级 LGD
                double lgdPool = (eadUncovered * lgdUncovered + eadCovered * lgdCovered) / eadTotal;

                // Step 7: 赋值给池内每笔借据
                for (AssetInput asset : poolAssets) {
                    asset.setLgdValue(lgdPool);
                }
            }

            // 无押品的借据：退化为 LGD_uncovered 路径
            for (AssetInput asset : customer.getAssets()) {
                if (asset.getLgdValue() == null) {
                    String key = key(asset.getGroupId(), "NONE", asset.getProductType());
                    double lgd = lgdCurveMap.getOrDefault(key, defaultLgd);
                    asset.setLgdValue(lgd);
                    asset.setLgdException("WARN");
                }
            }
        }
    }
}
```

### 7.3 ECL 计算

**输入字段：**

| 计算输入项 | 来源 | 字段名 |
|------------|------|--------|
| `ead_total` | 6.4 EAD 计算引擎输出 | `total_ead` |
| `lgd_value` | 6.5 LGD 计算引擎输出 | `lgd_pool` 赋值到资产级 `lgd_value` |
| `pd_details` | 6.3 PD 计算引擎输出 | `pd_details` |
| `pd_value` | 6.3 PD 计算引擎输出 | `pd_details[].pd_value` |
| `scenario_weight` | 6.3 PD 计算引擎输出 | `pd_details[].weight` |

**输出**：ecl_weighted + ecl_details（多情景 ECL JSON）

**设计说明：**
- 纯公式计算，无条件匹配
- PD 值已由 6.3 按 stage_result 完成转换（Stage 1=12M / Stage 2=永续 / Stage 3=100%），7.3 不重算
- 情景权重由 7.3 统一加权，6.3 只透传 weight
- `EAD` 使用 6.4 输出的资产级 `total_ead`，即 `on_bs_ead + off_bs_ead`
- `LGD` 使用 6.5 输出到资产级的 `lgd_value`；该值由押品池级 `LGD_pool` 赋值而来，7.3 不复用 6.5 的池级 `EAD_total`
- 若前序 PD / EAD / LGD 任一关键结果缺失或存在阻断异常，7.3 不计算该资产 ECL
- 运行时校验 `pd_details[].weight` 合计应为 1.0；不满足则标记参数异常并阻断该资产
- 金额计算使用高精度 Decimal，中间结果不提前四舍五入；单情景 ECL 和加权 ECL 输出保留 2 位小数

**核心公式：**

```
For each scenario i in pd_details:
  ECL_i = EAD × pd_value_i × LGD

ecl_weighted = Σ(ECL_i × weight_i)
```

**核心逻辑：**

```java
public class EclCalculator {

    @Override
    public void execute(JobContext ctx) {
        for (CustomerContext customer : ctx.getCustomers()) {
            for (AssetInput asset : customer.getAssets()) {
                if (asset.hasBlockingException()
                        || asset.getPdDetails() == null || asset.getPdDetails().isEmpty()
                        || asset.getTotalEad() == null
                        || asset.getLgdValue() == null) {
                    asset.setEclException("ECL_CALC_BLOCKED");
                    continue;
                }

                double ead = asset.getTotalEad();
                double lgd = asset.getLgdValue();
                List<PdDetail> pdDetails = asset.getPdDetails();

                double weightSum = pdDetails.stream()
                    .mapToDouble(PdDetail::getWeight)
                    .sum();
                if (Math.abs(weightSum - 1.0) > 0.000001) {
                    asset.setEclException("ECL_PD_WEIGHT_INVALID");
                    continue;
                }

                List<EclDetail> eclDetails = new ArrayList<>();
                double eclWeighted = 0;

                // Step 1: 按情景计算 ECL
                for (PdDetail pd : pdDetails) {
                    // pd_value 已是 6.3 按 stage 转换后的值
                    double ecl = ead * pd.getPdValue() * lgd;
                    eclDetails.add(new EclDetail(pd, ecl));
                    eclWeighted += ecl * pd.getWeight();
                }

                asset.setEclDetails(roundScenarioAmounts(eclDetails, 2));
                asset.setEclWeighted(round(eclWeighted, 2));
            }
        }
    }
}
```

**设计说明：**
- 规则一次性加载到内存，用 ConditionEvaluator 解析 conditions JSON（与 6.2 一致）
- 多条规则命中时，只选等效调整比例最高的 1 条（非累加）；等效比例相同时取 priority 最小
- 叠加基数固定为 EAD，作用于 ECL_FINAL
- 规则需满足 `effective_date <= calc_date` 且 `expiry_date` 为空或不早于 `calc_date`
- `conditions` 仅使用当前源数据可提供字段，如 `industry_cn`、`product_type`、`customer_no`、`is_npl`、`overdue_days`、`loan_classif_cd`、`guarantee_type`、`crr_final`、`ext_rating_this_year`
- EAD <= 0 时，FIXED 规则不参与等效比例比较，避免除零；无其他可选规则时不产生叠加金额

**核心逻辑：**

```java
public class OverlayEngine implements EclEngine {

    private List<OverlayRule> rules;  // 一次性加载

    @Override
    public void execute(JobContext ctx) {
        String schemeId = ctx.getSchemeId();
        this.rules = loadRules(schemeId);

        for (CustomerContext customer : ctx.getCustomers()) {
            for (AssetInput asset : customer.getAssets()) {
                processAsset(asset);
            }
        }
    }

    private void processAsset(AssetInput asset) {
        // Step 1: 按 group_id 过滤 + conditions 匹配
        List<OverlayRule> matched = new ArrayList<>();
        for (OverlayRule rule : rules) {
            if (!rule.getGroupId().equals(asset.getGroupId())) continue;
            if (!isEffective(rule, asset.getCalcDate())) continue;
            if (ConditionEvaluator.evaluate(rule.getConditions(), asset.toFieldMap())) {
                matched.add(rule);
            }
        }

        if (matched.isEmpty()) {
            asset.setEclOverlayTotal(0);
            asset.setEclFinal(asset.getEclWeighted());
            return;
        }

        // Step 2: 计算各规则的等效比例，选最高
        OverlayRule selected = null;
        double maxRatio = -1;

        for (OverlayRule rule : matched) {
            double ratio = switch (rule.getAdjustmentType()) {
                case ADDBP -> rule.getAdjustmentValue() / 10000.0;
                case PERCENTAGE -> rule.getAdjustmentValue();
                case FIXED -> asset.getTotalEad() > 0
                    ? rule.getAdjustmentValue() / asset.getTotalEad()
                    : -1;
            };
            if (ratio < 0) continue;
            if (ratio > maxRatio
                    || (ratio == maxRatio && selected != null
                        && rule.getPriority() < selected.getPriority())) {
                maxRatio = ratio;
                selected = rule;
            }
        }

        if (selected == null) {
            asset.setEclOverlayTotal(0);
            asset.setEclFinal(asset.getEclWeighted());
            return;
        }

        // Step 3: 算叠加金额
        double overlay = switch (selected.getAdjustmentType()) {
            case ADDBP -> asset.getTotalEad() * (selected.getAdjustmentValue() / 10000.0);
            case PERCENTAGE -> asset.getTotalEad() * selected.getAdjustmentValue();
            case FIXED -> selected.getAdjustmentValue();
        };

        // Step 4: 输出
        asset.setEclOverlayTotal(overlay);
        asset.setEclFinal(asset.getEclWeighted() + overlay);
        asset.setSelectedOverlayId(selected.getRuleId());
        asset.setEffectiveRatio(maxRatio);
    }
}
```

## 11. 非功能设计

### 10.1 性能

| 指标 | 目标 | 措施 |
|------|------|------|
| 批量计算 | 100万笔 ≤ 2 小时 | 并行引擎 + 分批提交 |
| API 响应 | P99 ≤ 3s | Redis 缓存 + 异步任务 |
| 并发 | 5 任务同时 | 线程池隔离 + 防重锁 |

**线程池**：core=8, max=16, 每批 5000 条, 每 10000 条提交

### 10.2 缓存

- 参数缓存：Redis，按 `scheme_id` 分区缓存，TTL=1 小时
- ECL 方案发布时清除对应 `scheme_id` 的缓存
- 试算查询：不写缓存（避免污染正式缓存）

### 10.3 异常处理体系

| 错误码 | 说明 | 级别 |
|--------|------|------|
| ECL_001 | PD查询无结果 | BLOCKING（阻断） |
| ECL_002 | EAD为负数 | WARN（强制为0） |
| ECL_003 | 无押品 | WARN（退化为无押品路径） |
| ECL_004 | 无生效方案 | BLOCKING |
| ECL_005 | 对账校验失败 | BLOCKING |
| ECL_006 | 方案状态不允许操作 | BLOCKING（如修改已生效方案） |

### 10.4 扩展性

- **情景可扩展**：tbl_pd_scenario 表支持动态添加情景
- **引擎 SPI**：新引擎实现 `EclEngine` 接口即可接入调度器
- **参数模块**：新参数类型只需添加以 `scheme_id` 为 FK 的表，天然支持版本控制
- **ECL 方案复制**：新模块加入时，只需在复制脚本中添加对应表的复制逻辑

## 附录：关键业务规则速查

| 规则 | 逻辑 |
|------|------|
| 空值通配 | 明细字段 NULL → 通用匹配 |
| 源数据空值 | 源数据字段为空且参数规则非空 → 不命中 |
| 优先级命中 | 多条命中取 priority 最小 |
| 兜底分组 | 无匹配 → GRP_DEFAULT |
| 4 维度全 NULL 禁止 | 禁止保存全通配明细 |
| PD 无兜底 | 查不到 pd_value → 阻断 |
| LGD 无兜底 | 查不到 lgd_base_value → 先查 NONE 路径，仍未找到 → 使用方案 default_lgd |
| CCF 无兜底 | 查不到 ccf_value → 使用方案 default_ccf（默认 0=不增加表外 EAD）|
| EAD 负数 | 强制为 0 |
| 无押品 LGD | 退化为 collateral_type='NONE' 的 lgd_base_value 路径 |
| EAD 折现率 | 有还款计划时按方案 discount_rate 折现未来现金流 |
| 方案缺省参数 | discount_rate / default_ccf / default_lgd / lgd_floor 在方案总览页配置，跟随方案版本 |
| 叠加非累加 | 多条规则 → 取等效比例最高 1 条，非累加 |
| 叠加基数 EAD | 叠加金额 = EAD × 调整比例 |
| 叠加目标 ECL_FINAL | 叠加金额直接加到最终 ECL 上，不修改中间参数 |
| 对账校验 | abs(6.8 - 6.7) ≥ 0.01 → 阻断 |
| **DRAFT 新建/复制** | 新建或从生效方案复制后，方案状态 = DRAFT，可自由修改 |
| **EFFECTIVE 冻结** | EFFECTIVE 方案下所有参数只读，不可修改 |
| **阶段判断顺序** | 先 FORWARD 向前判定（按 priority）→ 再比较 last_stage → 改善才查 ROLLBACK |
| **回跳限制** | Stage 3→1 禁止；Stage 3→2 需观察期+还本付息；Stage 2→1 需评级回升+逾期消除 |
| **兜底阶段** | FORWARD 无匹配 → Stage 1，exception_flag = 'Y' |
| **回跳失败** | 不满足 ROLLBACK 规则 → 保持原阶段，exception_flag = 'Y' |
| **立即生效** | 发布时直接当场将方案切换为 EFFECTIVE |
| **计划生效** | 发布时设置 effective_date，到时间由批处理自动生效 |
| **生效批处理** | 计划生效：PUBLISHED + effective_date <= TODAY → EFFECTIVE；原方案 → EXPIRED |
| **正式跑批取生效方案** | 自动获取唯一 EFFECTIVE 方案 |
| **试算指定方案** | 支持任意指定方案 ID 重算 |
