# ECL 减值计量系统 v3.1 — 全量开发实现计划

> **面向 AI 代理的工作者：** 推荐使用 subagent-driven-development 逐阶段实现本计划。
> 步骤使用复选框（`- [ ]`）语法来跟踪进度。

**目标：** 按技术设计说明书，实现 ECL 减值计量系统 v3.1 的全部五层架构（数据层→参数层→引擎层→计算层→报表层）+ REST API + React 前端。

**架构：** Spring Boot 3.x 单体分层架构，Maven 多模块聚合，MySQL 生产 / H2 开发测试，MyBatis-Plus ORM。

**技术栈：** Java 21, Spring Boot 3.x, MyBatis-Plus, MySQL 8.x, H2, Liquibase, Redis/Redisson, MapStruct, Spring Doc OpenAPI, JUnit5+Mockito, React 18+TS+Vite+Ant Design 5+ECharts

### 阶段 2：参数层 — ECL 方案管理（5.1）

**目标：** 实现 ECL 方案的 CRUD、复制、发布、状态机、对比功能。

- [ ] **任务 2.1：EclSchemeService 核心接口 + 实现**

  **创建：** `ecl-parameter/.../scheme/EclSchemeService.java`
  ```java
  public interface EclSchemeService {
      EclSchemeVO createScheme(SchemeCreateReq req);
      EclSchemeVO copyFromEffective(String description);
      EclSchemeVO publishScheme(String schemeId, boolean immediate, LocalDate effectiveDate);
      EclSchemeVO getScheme(String schemeId);
      List<EclSchemeVO> listSchemes(String status);
      EclSchemeVO getEffectiveScheme();
      List<SchemeDiffVO> compareSchemes(String schemeId1, String schemeId2);
      void expireScheme(String schemeId);
  }
  ```

  实现要点：
  - `createScheme`：生成 UUID+业务编码(SCH_序列)，status=DRAFT，scheme_version=v1.0
  - `copyFromEffective`：查找当前 EFFECTIVE 方案 + 全量复制（调用 SchemeCopyService）
  - `publishScheme`：状态机校验（DRAFT→PUBLISHED），immediate=true 则直接→EFFECTIVE
  - `getEffectiveScheme`：查询 status=EFFECTIVE 的唯一记录
  - `compareSchemes`：遍历所有参数模块，对比行数/差异（先返回 module 级对比）

- [ ] **任务 2.3：SchemeCopyService（复制机制）**

  **创建：** `ecl-parameter/.../copy/SchemeCopyService.java`

  从源 scheme_id 复制全部参数表到目标 scheme_id：
  1. 复制 tbl_risk_group（新 group_id + 新 scheme_id）
  2. 建立旧 group_id → 新 group_id 映射
  3. 复制 tbl_risk_group_detail（用映射后的 group_id）
  4. 复制 tbl_stage_rule（用映射后的 group_id）
  5. 复制 tbl_crr_rating_drop_rule（用映射后的 group_id）
  6. 复制 tbl_pd_scenario（保留原 scenario_id）
  7. 复制 tbl_pd_curve（用映射后的 group_id）
  8. 复制 tbl_lgd_curve（用映射后的 group_id）
  9. 复制 tbl_lgd_collateral_discount
  10. 复制 tbl_lgd_depreciation
  11. 复制 tbl_ccf_curve
  12. 复制 tbl_overlay_rule（用映射后的 group_id）

### 阶段 10：引擎层 — 各引擎实现

- [ ] **任务 10.1：RiskGroupEngine（6.1）**

  **创建：** `ecl-engine/.../riskgroup/RiskGroupEngine.java`

  实现 EclEngine 接口：
  1. 一次性加载所有 tbl_risk_group + tbl_risk_group_detail（按 scheme_id）
  2. 按 priority 排序
  3. 对每个 AssetInput，用 ConditionEvaluator 按 6 维字段匹配
  4. 匹配成功 → asset.groupId = 匹配的 group_id；失败 → "GRP_DEFAULT"

- [ ] **任务 10.2：StageEngine（6.2）**

  **创建：** `ecl-engine/.../stage/StageEngine.java`

  两步法：
  1. FORWARD 判定（按 priority）→ 暂存 stage_to
  2. 如果 stage_to 比 last_stage 改善 → 查 ROLLBACK 规则
  3. 判断顺序：Stage 3 → Stage 2 → Stage 1
  4. 兜底：无匹配 → Stage 1，exception_flag='Y'

- [ ] **任务 10.3：PdEngine（6.3）**

  **创建：** `ecl-engine/.../pd/PdEngine.java`

  1. 按 (group_id, rating_code, scenario_id) 查 tbl_pd_curve
  2. 按 stage_result 转换 PD 值：Stage 1=12M PD, Stage 2=永续 PD, Stage 3=100%
  3. 多情景返回 List<PdDetail>（含 weight）
  4. 查不到 → BLOCKING 异常

- [ ] **任务 10.4：EadEngine（6.4）**

  **创建：** `ecl-engine/.../ead/EadEngine.java`

  1. EAD = 表内余额 + 表外敞口 × CCF
  2. 查 tbl_ccf_curve → 按 product_type, commitment_type, commitment_days 区间匹配
  3. 查不到 CCF → 用 scheme.default_ccf
  4. EAD 负值 → 强制为 0

- [ ] **任务 10.5：LgdEngine（6.5）**

  **创建：** `ecl-engine/.../lgd/LgdEngine.java`

  1. 按 (group_id, collateral_type, product_type) 查 tbl_lgd_curve
  2. 有押品 → LGD = lgd_base_value × (1 - discount_rate)，押品折旧按年折算
  3. 无押品 → 查 collateral_type='NONE' 的 lgd_base_value
  4. 全查不到 → 用 scheme.default_lgd
