## 1. 文档概述

### 1.1 背景

本文档为北京银行 ECL 减值计量系统 v3.1 的技术设计说明书，依据《ECL_v3.1_业务需求说明书》编制。系统遵循 IFRS 9 / CAS 22 准则，实现全行信用风险资产的预期信用损失减值计量。

### 1.2 范围

本文档覆盖五层架构（数据层 → 参数层 → 引擎层 → 计算层 → 报表层）的全量技术设计，包括：

- 系统架构与技术选型
- 各模块的包结构、类设计、核心流程
- 7+1 引擎的详细设计与核心伪码
- 数据库表设计（以 ECL 减值方案为主线的统一版本控制）
- REST API 接口定义
- 非功能需求设计（性能/安全/可扩展）

### 1.3 术语对照

| BRD 术语 | 技术设计对应 |
|----------|------------|
| 借据（asset） | `Asset` / `LoanFacility` |
| 客户 | `Customer` |
| 押品池 | `CollateralPool` |
| 风险分组 | `RiskGroup` |
| 方案（scheme） | 顶层 `EclScheme`，全局版本控制单元 |
| ECL 减值方案 | `tbl_ecl_scheme`，所有参数模块的版本根 |
| 参数层 | `parameter` 模块 |
| 引擎层 | `engine` 模块 |

### 1.4 ID 生成规则

- **主键 ID**：统一使用 UUID，保证全局唯一
- **业务编码**：每张业务表额外提供人工可读编码字段（如 `scheme_code` / `group_code`），用于页面展示和模型人员识别
- **编码示例**：`SCH_001`、`GRP_001`、`JOB_20260618_001`
- **编码规则**：ID 不对外展示，前端和报表统一展示业务编码

## 5. 参数层详细设计

### 5.1 参数层总体设计

#### 5.1.1 设计哲学：以 ECL 减值方案为顶层管理单元

参数层采用 **"ECL 减值方案统一管理 + 各模块明细按 scheme_id 隔离"** 的架构。

核心变化（对比原 BRD）：

| 原设计 | 新设计 | 原因 |
|--------|--------|------|
| 各模块各自维护方案表（6 套方案/状态机） | 统一由 `tbl_ecl_scheme` 管理版本和状态 | 1 套状态机 vs 6 套，运维成本低 |
| 以 `tbl_risk_group` 作为全局主表关联 | 以 `tbl_ecl_scheme` 为顶层，各模块挂载其下 | CCF/押品折扣等不依赖 group_id 的表挂靠自然 |
| 各模块方案独立发布 | ECL 方案统一定版、发布、生效 | 模型人员操作的是"一套方案"而非散装参数 |

#### 5.1.2 纯方案级版本控制

所有参数变更统一通过 ECL 方案版本管理，**不再有行级状态和行级生效时间**。

**核心原则：**

```
变更的唯一路径：复制方案 → DRAFT 修改 → 整体发布

不存在"在已生效方案内单行修改"的路径
不存在行级 row_status / effective_date
所有明细数据跟随所属 ECL 方案的版本状态
```

#### 5.1.3 两种变更场景

##### 场景一：全新创建 ECL 方案（从零开始）

```
新建 ECL 方案 → scheme_status = DRAFT
  └─ 新增各参数层数据
       （自由修改）
  ↓ 发布
  ├─ [立即生效] → ECL 方案直接 → EFFECTIVE
  └─ [计划生效] → 设置 effective_date → PUBLISHED
                   ↓ 到达生效日期（批处理 01:00）
                  ECL 方案 → EFFECTIVE
```

##### 场景二：整体调整 ECL 方案（全部复制）

```
SCH_001（EFFECTIVE）
  │ [模型人员点击"新建方案"]
  ▼
复制 SCH_001 全部数据 → SCH_002（新 scheme_id，版本号+1）
  │ SCH_001 不受影响，仍在正式跑批中使用
  │ 方案状态 = DRAFT
  │ [模型人员在新方案上自由修改]
  ▼
SCH_002（DRAFT）
  │ 可增删改任意参数
  │ [发布]
  ├─ [立即生效] → SCH_002 + 所有数据直接 → EFFECTIVE
  │                 SCH_001 + 所有数据直接 → EXPIRED
  └─ [计划生效] → 设置 effective_date
                   SCH_002 → PUBLISHED
                   ↓ 到达生效日期
                   SCH_002 → EFFECTIVE
                   SCH_001 → EXPIRED
```

#### 5.1.4 方案状态机

```java
public enum SchemeStatus {
    DRAFT("草稿"),
    PUBLISHED("已发布"),
    EFFECTIVE("已生效"),
    EXPIRED("已失效");

    public boolean canTransitionTo(SchemeStatus target) {
        return switch (this) {
            case DRAFT -> target == PUBLISHED;
            case PUBLISHED -> target == EFFECTIVE;
            case EFFECTIVE -> target == EXPIRED;
            case EXPIRED -> false;
        };
    }
}
```

#### 5.1.5 自动生效批处理（每日 01:00）

仅作用于选择"计划生效"的方案，"立即生效"的方案已当场处理完毕。

```
1. 扫描所有 PUBLISHED 且 effective_date <= TODAY 的 ECL 方案
   → 方案状态改为 EFFECTIVE
2. 扫描被新方案替代的旧 EFFECTIVE 方案
   → 旧方案状态改为 EXPIRED
```

#### 5.1.6 复制机制

当模型人员创建新方案时，系统自动执行全量复制：

```
1. 创建 tbl_ecl_scheme 记录（新 scheme_id, DRAFT, 版本号+1）
2. 复制 tbl_risk_group（关联新 scheme_id）
3. 复制 tbl_risk_group_detail
4. 复制 tbl_stage_rule
5. 复制 tbl_crr_rating_drop_rule
6. 复制 tbl_pd_scenario
7. 复制 tbl_pd_curve
8. 复制 tbl_lgd_curve
9. 复制 tbl_lgd_collateral_discount
10. 复制 tbl_lgd_depreciation
11. 复制 tbl_ccf_curve
12. 复制 tbl_overlay_rule
```

复制后整个方案处于 DRAFT 状态，模型人员可在新方案上自由修改任意参数。
原 ECL 方案不受影响，继续在正式跑批中使用。

### 5.2 风险分组参数（5.2）

#### 5.2.1 设计思路

风险分组采用 **"主数据 + 明细"两表模式**（去掉了原方案的方案表，版本控制统一由 `tbl_ecl_scheme` 管理）。

- `tbl_risk_group`：分组主数据，带 `scheme_id`，不同 ECL 方案可以有不同数量和定义的分组
  - 纯元数据（名称、排序、描述），不参与版本控制
  - 删除分组只需在 DRAFT 方案内删除对应行，老方案数据不受影响
- `tbl_risk_group_detail`：6 维匹配规则明细，带 `scheme_id` + `group_id`
- 匹配规则变化不影响 `group_id`，因此 PD/LGD/Stage/Overlay 等引用 `group_id` 的表不受影响

#### 5.2.2 分组匹配算法（核心逻辑）

```java
public String matchGroup(AssetInput asset, String schemeId) {
    List<RiskGroupDetail> details = loadDetails(schemeId);
    
    for (RiskGroupDetail d : details) {
        if (matches(asset, d)) return d.getGroupId();
    }
    return "GRP_DEFAULT";  // 兜底
}

private boolean matches(AssetInput asset, RiskGroupDetail d) {
    return matchField(asset.getBusinessLine(), d.getBusinessLine())
        && matchField(asset.getCustomerType(), d.getCustomerType())
        && matchField(asset.getProductType(), d.getProductType())
        && matchField(asset.getIndustryCode(), d.getIndustryCode())
        && matchField(asset.getRegionCode(), d.getRegionCode())
        && matchField(asset.getCollateralType(), d.getCollateralType());
}

// NULL = 通配
private boolean matchField(String assetValue, String ruleValue) {
    return ruleValue == null || ruleValue.isEmpty() || ruleValue.equals(assetValue);
}
```

### 5.3 阶段划分参数（5.3）

#### 5.3.1 阶段状态机

阶段划分本质上是一个三态状态机，按 IFRS 9 / CAS 22 准则将资产划分为三个阶段：

```
            ┌──────────────────────────────────────┐
            │                                      │
            ▼                                      │
        ┌────────┐   信用风险显著增加   ┌────────┐  │
   ───▶ │Stage 1 │ ─────────────────▶ │Stage 2 │  │
        │ 正常类  │ ◀───────────────── │ 关注类  │  │
        └────────┘  风险消除+观察期满足 └────────┘  │
            ▲                                  │    │
            │ 禁止直接回跳                      │    │
            │ （须经 Stage 2 中转）             ▼    │
            │                              ┌────────┐
            │         6个月观察期+还本付息   │Stage 3 │ ◀── 违约触发
            │         ┌─────────────────── │ 损失类  │
            │         │                    └────────┘
            │         │                         │
            │         ▼                         │
            │      ┌────────┐                   │
            └───── │Stage 2 │ ◀─────────────────┘
        （不允许）   │ 关注类  │   风险解除
                   └────────┘
```

**关键约束：**
- ❌ Stage 3 → Stage 1 禁止
- ✅ Stage 3 → Stage 2 → Stage 1（须经观察期中转）
- ✅ Stage 2 → Stage 3（直接，无中间过渡）

#### 5.3.2 表设计

阶段划分采用 **两条规则表** 的设计：

- `tbl_stage_rule`：存储 FORWARD（向前判定）和 ROLLBACK（回跳校验）两类规则
  - 触发条件使用 **JSON 字段** 存储，不同风险分组可定义不同的条件结构
  - 不再使用散列的 trigger 字段（避免稀疏表）
- `tbl_crr_rating_drop_rule`：CRR / 国际评级下降阈值表，供 FORWARD 和 ROLLBACK 规则复用


#### 5.3.4 取值逻辑（核心算法）

```
输入：
  group_id
  last_stage          ← 借据的当前阶段（上期计算结果）
  overdue_days, crr_rating, five_category, default_flag, media...（借据字段）
  scheme_id           ← 绑定的 ECL 方案

═══ Step 1 — 向前判定（FORWARD） ═══

1. 查询 tbl_stage_rule →
   WHERE scheme_id = :scheme_id
     AND group_id = :group_id
     AND rule_type = 'FORWARD'
   ORDER BY priority ASC

2. 遍历规则，逐条解析 conditions JSON 匹配：
   ├── 规则 A JSON 中的条件全部满足 → stage_to = 该规则的目标
   │     → 进入 Step 2（回跳校验）
   └── 所有规则都不匹配 → stage_to = STAGE_1
        exception_flag = 'Y'
        → 进入 Step 2

═══ Step 2 — 回跳校验（ROLLBACK） ═══

3. 判断是否需要回跳校验：
   如果 stage_to < last_stage（阶段改善，如 3→2 或 2→1 或 3→1）
      → 需校验
   否则（阶段不变或恶化）
      → 直接返回 stage_to

4. 查询 ROLLBACK 规则：
   WHERE scheme_id = :scheme_id
     AND group_id = :group_id
     AND rule_type = 'ROLLBACK'
     AND stage_from = :last_stage
     AND stage_to = :stage_to

5. 匹配规则中的 conditions JSON：
   ├── 满足条件 → 返回 stage_to（允许回跳）
   └── 不满足条件 → 返回 last_stage（保持原阶段，禁止回跳）
        exception_flag = 'Y'
```

#### 5.3.5 引擎层伪码

```java
public StageResult determineStage(AssetInput asset, String schemeId, Stage lastStage) {
    // Step 1 — 向前判定
    List<StageRule> forwardRules = stageRuleMapper.selectList(
        new LambdaQueryWrapper<StageRule>()
            .eq(StageRule::getSchemeId, schemeId)
            .eq(StageRule::getGroupId, asset.getGroupId())
            .eq(StageRule::getRuleType, "FORWARD")
            .orderByAsc(StageRule::getPriority)
    );

    Stage targetStage = Stage.STAGE_1;  // 兜底
    boolean exceptionFlag = true;
    String matchedTriggerType = null;

    for (StageRule rule : forwardRules) {
        if (matchConditions(rule.getConditions(), asset)) {
            targetStage = rule.getStageTo();
            matchedTriggerType = extractTriggerType(rule.getConditions());
            exceptionFlag = false;
            break;
        }
    }

    // Step 2 — 回跳校验
    if (targetStage.ordinal() < lastStage.ordinal()) {
        // 阶段改善，需要回跳校验
        List<StageRule> rollbackRules = stageRuleMapper.selectList(
            new LambdaQueryWrapper<StageRule>()
                .eq(StageRule::getSchemeId, schemeId)
                .eq(StageRule::getGroupId, asset.getGroupId())
                .eq(StageRule::getRuleType, "ROLLBACK")
                .eq(StageRule::getStageFrom, lastStage.name())
                .eq(StageRule::getStageTo, targetStage.name())
        );

        boolean rollbackAllowed = false;
        for (StageRule rule : rollbackRules) {
            if (matchConditions(rule.getConditions(), asset)) {
                rollbackAllowed = true;
                break;
            }
        }

        if (!rollbackAllowed) {
            // 禁止回跳，保持原阶段
            return new StageResult(lastStage, "ROLLBACK_BLOCKED", true);
        }
    }

    return new StageResult(targetStage, matchedTriggerType, exceptionFlag);
}

/** 解析 JSON conditions 并匹配资产字段 */
private boolean matchConditions(JSONObject conditions, AssetInput asset) {
    // 通用逻辑引擎（支持 overdue_days / five_category / crr_drop / media_sentiment / default 等）
    return ConditionEvaluator.evaluate(conditions, asset);
}
```

### 5.4 PD 参数（5.4）

#### 5.4.1 设计

PD（违约概率）是 ECL 公式 `ECL = PD × LGD × EAD` 的核心参数。每笔借据根据其 `group_id` + `rating_code` 从 PD 曲线表获取对应值。

**表设计：**

- `tbl_pd_scenario`：情景定义表（乐观/基准/悲观），`scenario_type` 为 **自由文本**，支持未来扩展压力情景等新情景
- `tbl_pd_curve`：PD 曲线表，按 `scheme_id + group_id + scenario_id + rating_code` 组织 12M PD 值

**核心特点：**

| 特性 | 设计 |
|------|------|
| 全行方案数 | 1 个（由 tbl_ecl_scheme 统一管理） |
| 分组差异化 | 通过 `tbl_pd_curve.group_id` 隔离 |
| 情景扩展性 | scenario_type 自由文本，新增情景只需加行 |
| 权重约束 | 同一方案下所有情景 weight 之和 = 1.0，发布前校验 |
| 兜底 | **无兜底**，缺值即阻断 |

**初始数据：**

| scenario_type | scenario_name | weight |
|--------------|--------------|--------|
| OPTIMISTIC | 乐观情景 | 0.20 |
| BASELINE | 基准情景 | 0.60 |
| PESSIMISTIC | 悲观情景 | 0.20 |

#### 5.4.2 取值逻辑

```
输入：scheme_id + group_id + rating_code + stage_result
    ↓
1. 查 tbl_pd_scenario → scheme_id
   → 获取所有情景的 scenario_id + weight（如 3 行）
    ↓
2. 查 tbl_pd_curve → scheme_id + group_id + rating_code + 各 scenario_id
   → 返回该评级在每个情景下的 pd_value（多行）
    ↓
3. 加权计算 12M PD：
   PD_12M_weighted = Σ(pd_value_i × weight_i)
    ↓
4. 若为 Stage 2/3，计算存续期 PD（在引擎层计算）：
   PD_Lifetime = 1 - (1 - PD_12M)^(remaining_months / 12)
    ↓
异常：Step 2 无结果 → 阻断（PD 无兜底）
```

#### 5.4.3 引擎层伪码

```java
public PdResult loadPd(String schemeId, String groupId, String ratingCode, int remainingMonths) {
    // Step 1: 查情景
    List<PdScenario> scenarios = pdScenarioMapper.selectList(
        new LambdaQueryWrapper<PdScenario>()
            .eq(PdScenario::getSchemeId, schemeId)
    );
    if (scenarios.isEmpty()) {
        throw new EclException("ECL_004", "方案无 PD 情景配置");
    }

    // Step 2: 查 PD 曲线
    double pdWeighted = 0.0;
    for (PdScenario s : scenarios) {
        PdCurve curve = pdCurveMapper.selectOne(
            new LambdaQueryWrapper<PdCurve>()
                .eq(PdCurve::getSchemeId, schemeId)
                .eq(PdCurve::getGroupId, groupId)
                .eq(PdCurve::getRatingCode, ratingCode)
                .eq(PdCurve::getScenarioId, s.getScenarioId())
        );
        if (curve == null) {
            throw new EclException("ECL_001",
                "PD查询无结果: group_id=" + groupId + ", rating_code=" + ratingCode);
        }
        pdWeighted += curve.getPdValue() * s.getWeight();
    }

    // Step 3: 存续期 PD（引擎层累积）
    double pdLifetime = 1 - Math.pow(1 - pdWeighted, remainingMonths / 12.0);

    return new PdResult(pdWeighted, pdLifetime);
}
```

### 5.5 LGD 参数（5.5）

#### 5.5.1 设计

LGD（违约损失率）是 ECL 公式 `ECL = PD × LGD × EAD` 的第二个核心参数。定义为违约时损失金额与 EAD 的比例。

**表设计（3 张表）：**

| 表 | 作用 | 是否依赖 group_id |
|----|------|-----------------|
| `tbl_lgd_curve` | LGD 基准值表，按 `(group_id, collateral_type, product_type)` 组合细分 | ✅ |
| `tbl_lgd_collateral_discount` | 押品折扣率（押品市场属性，全方案共享） | ❌ |
| `tbl_lgd_depreciation` | 押品随年限折旧率（押品物理属性，全方案共享） | ❌ |

**关键特性：**
- LGD 查不到 lgd_base_value 时：依次查 NONE 路径 → 使用方案级 default_lgd
- 与 PD 不同（PD 仍为阻断，无兜底）

**与 PD 的关键区别：**

| | PD | LGD |
|--|----|-----|
| 情景 | ✅ 三情景加权（乐观/基准/悲观） | ❌ 单一基准值 |
| 细分维度 | group_id + rating_code | group_id + collateral_type + product_type |
| 兜底 | ❌ 无兜底，阻断 | ✅ 可配置缺省值（方案 default_lgd，NONE 路径仍优先） |

#### 5.5.2 核心公式

```
对押品池内的每个押品：
  a. 查 tbl_lgd_curve → (scheme_id, group_id, collateral_type, product_type)
     → 取 lgd_base_value
  b. 查 tbl_lgd_depreciation → (scheme_id, collateral_type, year_offset)
     → 取 depreciation_rate
  c. 查 tbl_lgd_collateral_discount → (scheme_id, collateral_type)
     → 取 discount_rate（可选，一些银行直接合并到 lgd_base_value 中）
  d. 该押品净值 = 评估价值 × (1 + depreciation_rate) × (1 - discount_rate)
  e. 该押品覆盖额 = min(押品净值, 池内对应借据的 EAD)

押品池汇总：
  EAD_covered    = Σ(各押品覆盖额)
  EAD_uncovered  = Σ(池内借据 EAD) - EAD_covered
  LGD_pool       = (EAD_covered × LGD_加权 + EAD_uncovered × LGD_uncovered) / Σ(池内借据 EAD)

  其中：
    LGD_加权     = Σ(各押品覆盖额 × 该押品的 lgd_base_value) / EAD_covered
    LGD_uncovered = 查 tbl_lgd_curve → (group_id, collateral_type='NONE', product_type)
```

#### 5.5.3 取值逻辑

```
输入：scheme_id + group_id + 押品池（各押品的 collateral_type + 评估价值）
       + product_type + 池内借据 EAD 汇总
    ↓
1. 查 tbl_lgd_curve → scheme_id + group_id + product_type
   → 获取该分组下各担保类型的 lgd_base_value（多行）
    ↓
2. 对押品池内每个押品：
   查 tbl_lgd_depreciation → scheme_id + collateral_type + 押品年龄
   → 获取该押品的折旧率
   查 tbl_lgd_collateral_discount → scheme_id + collateral_type
   → 获取该押品的折扣率
   该押品净值 = 评估价值 × (1 + depreciation_rate) × (1 - discount_rate)
   该押品覆盖额 = min(押品净值, 对应借据的 EAD)
   该押品的 LGD = lgd_base_value（从 Step 1 获取）
    ↓
3. 汇总押品池：
   EAD_covered = Σ 各押品覆盖额
   EAD_uncovered = 池内借据 EAD 汇总 - EAD_covered
   LGD_加权  = Σ(押品覆盖额 × lgd_base_value) / EAD_covered
   LGD_uncovered 查表获取（collateral_type = 'NONE' 或无押品时）
    ↓
4. LGD_pool = (EAD_covered × LGD_加权 + EAD_uncovered × LGD_uncovered) / 总 EAD
    ↓
异常：Step 1 查不到 lgd_base_value → 先查 collateral_type='NONE' 路径 → 仍未找到 → 使用方案 default_lgd
```

### 5.6 CCF 参数（5.6）

#### 5.6.1 设计

CCF（信用转换系数）用于将表外未提取额度转换为表内 EAD。**不依赖 group_id**，因为 CCF 是产品/承诺的监管属性，与业务分组无关。

**公式：** `EAD = 已提取余额 + 未提取额度 × CCF`

**表设计（1 张表）：**

| 表 | 唯一键 | 是否依赖 group_id |
|----|-------|-----------------|
| `tbl_ccf_curve` | `(scheme_id, product_type, commitment_type, commitment_days_min, commitment_days_max)` | ❌ |

**关键特性：**
- 无情景表（CCF 是监管标准值，加情景无业务意义）
- 兜底机制：查不到 ccf_value → 使用方案 default_ccf（可配置，默认 0=不额外增加表外 EAD）
- 与 5.4 PD（3 情景加权）差异化设计

#### 5.6.2 取值逻辑

```
输入：scheme_id + product_type + commitment_type + commitment_days
    ↓
查 tbl_ccf_curve → scheme_id + product_type + commitment_type
    + commitment_days BETWEEN commitment_days_min AND commitment_days_max
    → 得到 ccf_value
    ↓
异常：查不到 ccf_value → 使用方案 default_ccf（非阻断，用户可配置）
```

### 5.7 管理层叠加（5.7）

#### 5.7.1 设计

管理层叠加是 ECL 计算引擎输出后，管理层根据业务判断对特定范围进行人工干预调整的参数。

**表设计（1 张表）：**

```sql
    priority           INT            NOT NULL COMMENT '匹配优先级（越小越高）',
    conditions         JSON           NOT NULL COMMENT '匹配条件 JSON，支持按行业/地区/客户/产品等维度',
    effective_date     DATE           NOT NULL COMMENT '规则生效日期',
    expiry_date        DATE           COMMENT '规则失效日期',

    FOREIGN KEY (scheme_id, group_id) REFERENCES tbl_risk_group(scheme_id, group_id)
) COMMENT '管理层叠加规则表';
```

**核心规则：**
1. **基数固定为 EAD**：叠加金额 = EAD × 调整比例
2. **作用于 ECL_FINAL**：`ECL_final = ECL_calculated + 叠加金额`
3. **非累加**：多条规则命中时只选等效调整比例最高的 1 条
4. **匹配条件用 JSON**，与 5.3 阶段划分风格一致

#### 5.7.2 conditions JSON 约定

```json
-- 按行业范围匹配
{"industry_codes": ["J", "K", "L"]}

-- 按地区+行业复合匹配
{"region_codes": ["110000", "310000"], "industry_codes": ["J"]}

-- 按产品类型匹配
{"product_types": ["公司贷款", "银团贷款"]}

-- GROUP 范围（直接命中 group_id，不写 conditions）
-- 当 apply_scope 为 GROUP 时，conditions 可为空
```

#### 5.7.3 调整方式

| adjustment_type | 含义 | 叠加金额公式 |
|---------------|------|------------|
| ADDBP | 加基点 | EAD × (adjustment_value / 10000) |
| PERCENTAGE | 加百分比 | EAD × adjustment_value |
| FIXED | 固定金额 | adjustment_value（不依赖 EAD） |

#### 5.7.4 取值逻辑

```
输入：scheme_id + group_id + EAD + ECL_calculated + 借据维度字段
    ↓
1. 查 tbl_overlay_rule → scheme_id + group_id + 生效规则
   → 按 priority ASC 排序
    ↓
2. 逐条解析 conditions JSON 匹配
   → 收集所有命中的规则
    ↓
3. 计算各命中规则的等效调整比例：
   ADDBP：     等效比例 = adjustment_value / 10000
   PERCENTAGE：等效比例 = adjustment_value
   FIXED：     等效比例 = adjustment_value / EAD
    ↓
4. 选等效比例最高的 1 条（非累加）
   → 按选中的 adjustment_type 公式计算叠加金额
    ↓
5. ECL_final = ECL_calculated + 叠加金额
```

#### 5.7.5 `overlay_type` 说明

| overlay_type | 含义 | 影响计算 |
|-------------|------|---------|
| SPECIAL | 专项调整（如房地产行业专项） | ❌ 仅标签 |
| CALIBRATION | 模型校准调整 | ❌ 仅标签 |
| MODEL_ERROR | 模型误差补偿 | ❌ 仅标签 |

`overlay_type` 仅为分类标签，供前端筛选和审计追溯使用，不参与引擎计算逻辑。

### 6.1 风险分组引擎

**输入**：借据的 6 维属性（business_line + customer_type + product_type + industry_code + region_code + collateral_type）
**输出**：group_id + group_name + exception_flag

**设计说明：**
- 规则一次性加载到内存，后续每笔借据在内存中匹配，不逐笔查数据库
- 匹配逻辑使用通用的 `ConditionEvaluator` 组件，与 6.2 阶段划分、6.7 管理层叠加复用

**匹配逻辑：**

```java
public class RiskGroupingEngine implements EclEngine {
    
    @Override
    public void execute(JobContext ctx) {
        String schemeId = ctx.getSchemeId();
        this.rules = loadRules(schemeId);
        this.groupMap = loadGroupMap(schemeId);

        for (CustomerContext customer : ctx.getCustomers()) {
            for (AssetInput asset : customer.getAssets()) {
            String groupId = matchGroup(asset, rules, groupMap);
            asset.setGroupId(groupId);
            if ("GRP_DEFAULT".equals(groupId)) {
                asset.setGroupException("Y");
            }
        }
    }
    
    private String matchGroup(AssetInput asset) {
        for (RiskGroupDetail rule : rules) {
            // 6 维度 AND 匹配（NULL = 通配）
            // 使用 ConditionEvaluator 统一匹配
            if (matchRule(asset, rule)) {
                return rule.getGroupId();
            }
        }
        // 兜底
        return "GRP_DEFAULT";
    }
    
    private boolean matchRule(AssetInput asset, RiskGroupDetail rule) {
        return ConditionEvaluator.matchField(asset.getBusinessLine(), rule.getBusinessLine())
            && ConditionEvaluator.matchField(asset.getCustomerType(), rule.getCustomerType())
            && ConditionEvaluator.matchField(asset.getProductType(), rule.getProductType())
            && ConditionEvaluator.matchField(asset.getIndustryCode(), rule.getIndustryCode())
            && ConditionEvaluator.matchField(asset.getRegionCode(), rule.getRegionCode())
            && ConditionEvaluator.matchField(asset.getCollateralType(), rule.getCollateralType());
    }
}

// ConditionEvaluator —— 统一条件评估器，三个引擎共享
public class ConditionEvaluator {
    
    /** 字段匹配：NULL 或空 = 通配 */
    public static boolean matchField(String assetValue, String ruleValue) {
        if (ruleValue == null || ruleValue.isEmpty()) return true;  // 规则通配
        if (assetValue == null || assetValue.isEmpty()) return true;  // 资产通配
        return ruleValue.equals(assetValue);
    }
    
    /** JSON conditions 匹配（6.2、6.7 使用） */
    public static boolean evaluate(JSONObject conditions, Map<String, Object> input) {
        if (conditions == null || conditions.isEmpty()) return true;
        for (String key : conditions.keySet()) {
            Object condition = conditions.get(key);
            Object value = input.get(key);
            if (!evaluateField(condition, value)) return false;
        }
        return true;
    }
    
    private static boolean evaluateField(Object condition, Object value) {
        if (condition instanceof JSONObject) {
            JSONObject cond = (JSONObject) condition;
            if (cond.containsKey("min") || cond.containsKey("max")) {
                // 范围匹配（如 overdue_days: {min:31, max:90})
                return matchRange(cond, value);
            }
            if (cond.containsKey("in")) {
                // 枚举匹配（如 five_category: {in:["次级","可疑","损失"]})
                return matchIn(cond, value);
            }
        }
        // 精确匹配 / OR 逻辑
        return false;
    }
}
```

### 6.2 阶段划分引擎

**输入**：借据字段（逾期天数/评级/五级分类/违约标识/舆情等）+ group_id + last_stage
**输出**：stage_result + trigger_type + exception_flag

**设计说明：**
- 与 6.1 相同，规则一次性加载到内存，不逐笔查数据库
- 使用通用的 `ConditionEvaluator` 解析 JSON conditions
- 两步法：先 FORWARD 向前判定 → 若阶段改善再查 ROLLBACK 校验

**核心逻辑：**

```java
public class StageEngine implements EclEngine {

    @Override
    public void execute(JobContext ctx) {
        String schemeId = ctx.getSchemeId();

        // 一次性加载 FORWARD 和 ROLLBACK 规则
        List<StageRule> forwardRules = loadRules(schemeId, "FORWARD");
        List<StageRule> rollbackRules = loadRules(schemeId, "ROLLBACK");
        // 一次性加载 CRR 阈值映射
        Map<String, Integer> crrDropMap = loadCrrDropMap(schemeId);

        for (CustomerContext customer : ctx.getCustomers()) {
            for (AssetInput asset : customer.getAssets()) {
            StageResult result = determineStage(asset, forwardRules, rollbackRules, crrDropMap);
            asset.setStageResult(result);
            }
        }
    }

    private StageResult determineStage(AssetInput asset,
                                        List<StageRule> forwardRules,
                                        List<StageRule> rollbackRules,
                                        Map<String, Integer> crrDropMap) {

        // ═══ Step 1: FORWARD 向前判定 ═══
        // 顺序：先检视 Stage 3 → 再 Stage 2 → 最后 Stage 1（兜底）
        Stage targetStage = Stage.STAGE_1;
        String matchedTrigger = null;
        boolean exceptionFlag = true;

        for (StageRule rule : forwardRules) {
            if (ConditionEvaluator.evaluate(rule.getConditions(), asset.toFieldMap(), crrDropMap)) {
                targetStage = rule.getStageTo();
                matchedTrigger = extractTriggerType(rule.getConditions());
                exceptionFlag = false;
                break;
            }
        }

        // ═══ Step 2: ROLLBACK 回跳校验 ═══
        if (targetStage.ordinal() < asset.getLastStage().ordinal()) {
            // 阶段改善，需要回跳校验
            boolean passRollback = false;
            for (StageRule rule : rollbackRules) {
                if (rule.getStageFrom() == asset.getLastStage()
                        && rule.getStageTo() == targetStage) {
                    if (ConditionEvaluator.evaluate(rule.getConditions(), asset.toFieldMap())) {
                        passRollback = true;
                        break;
                    }
                }
            }

            if (!passRollback) {
                // 禁止回跳，保持原阶段
                return new StageResult(asset.getLastStage(), "ROLLBACK_BLOCKED", true);
            }
        }

        return new StageResult(targetStage, matchedTrigger, exceptionFlag);
    }
}

/** ConditionEvaluator 扩展——支持 CRR 查表 */
public class ConditionEvaluator {

    /** 基础字段匹配（6.1 使用） */
    public static boolean matchField(String assetValue, String ruleValue) { ... }

    /** JSON conditions 匹配（6.2、6.7 使用） */
    public static boolean evaluate(JSONObject conditions, Map<String, Object> input,
                                    Map<String, Integer> crrDropMap) {
        // ... 支持 overdue_days / five_category / crr_drop / media_sentiment / default 等
        for (String key : conditions.keySet()) {
            Object cond = conditions.get(key);
            if (key.equals("or")) {
                // OR 逻辑：任一子条件满足即通过
                JSONArray orItems = (JSONArray) cond;
                boolean anyMatch = false;
                for (Object item : orItems) {
                    if (evaluateSingle((JSONObject) item, input, crrDropMap)) {
                        anyMatch = true;
                        break;
                    }
                }
                if (!anyMatch) return false;
            } else if (key.equals("crr_drop") && crrDropMap != null) {
                // CRR 评级下降：查 tbl_crr_rating_drop_rule
                JSONObject crr = (JSONObject) cond;
                String currentRating = (String) input.get("crr_rating");
                int dropLevels = (int) input.get("rating_drop_levels");
                int threshold = crrDropMap.getOrDefault(currentRating, Integer.MAX_VALUE);
                if (dropLevels < threshold) return false;
            } else {
                if (!evaluateSingle(conditions, input, crrDropMap)) return false;
            }
        }
        return true;
    }
}
```

### 6.3 PD 计算引擎

**输入**：scheme_id + group_id + rating_code + stage_result + maturity_date + calc_date
**输出**：pd_details（JSON 数组，含各情景的 pd_value + weight）

**设计说明：**
- PD 曲线数据量小（~240 行），启动时一次性加载到内存 Map
- 纯查表 + 公式计算，无条件匹配，不需要规则引擎
- Stage 2 的永续 PD 在引擎层用生存公式计算

**核心逻辑：**

```java
public class PdEngine implements EclEngine {

    // 缓存：Map<(groupId, ratingCode, scenarioId), pdValue>
    private Map<String, Double> pdCache;

    @Override
    public void execute(JobContext ctx) {
        String schemeId = ctx.getSchemeId();

        // 一次性加载 PD 情景列表
        List<PdScenario> scenarios = loadScenarios(schemeId);
        // 一次性加载 PD 曲线到内存 Map
        this.pdCache = loadPdCurveMap(schemeId);

        for (CustomerContext customer : ctx.getCustomers()) {
            for (AssetInput asset : customer.getAssets()) {
            List<PdDetail> details = new ArrayList<>();

            for (PdScenario s : scenarios) {
                Double pdValue = pdCache.get(key(asset.getGroupId(), asset.getRatingCode(), s.getScenarioId()));

                if (pdValue == null) {
                    // PD 无兜底，阻断该笔借据
                    asset.setPdException("ECL_001");
                    continue;
                }

                // 按 stage_result 转换 PD 值
                double pd = convertPdByStage(pdValue, asset.getStageResult(),
                        asset.getMaturityDate(), ctx.getCalcDate());

                details.add(new PdDetail(s, pd));
            }

            asset.setPdDetails(details);
            }
        }
    }

    private double convertPdByStage(double pd12m, Stage stage, LocalDate maturity, LocalDate calcDate) {
        return switch (stage) {
            case STAGE_1 -> pd12m;                              // 12M PD，直接用
            case STAGE_2 -> {
                // 永续 PD：1 - (1 - PD_12M)^T
                double t = ChronoUnit.MONTHS.between(calcDate, maturity) / 12.0;
                if (t < 1) t = 1;                               // 不足 1 年按 1 年
                yield 1 - Math.pow(1 - pd12m, t);
            }
            case STAGE_3 -> 1.0;                                 // 全覆盖
        };
    }
}
```

### 6.4 EAD 计算引擎

**输入**：customer 下所有借据 + 额度信息
**输出**：每条借据的 on_bs_ead + off_bs_ead_allocated + total_ead

**设计说明：**
- EAD 是整个引擎链中最复杂的环节，涉及还款计划折现、表外 CCF 分配、跨借据分摊
- 以客户为中心，一次性处理该客户下的所有借据和额度
- CCF 参数一次性加载到内存

**核心逻辑：**

```java
public class EadEngine implements EclEngine {

    private Map<String, CcfValue> ccfCache;  // (productType, commitmentType) → CCF
    private double defaultCcf;                // 方案级 CCF 缺省值

    @Override
    public void execute(JobContext ctx) {
        String schemeId = ctx.getSchemeId();
        this.ccfCache = loadCcfMap(schemeId);
        this.defaultCcf = ctx.getDefaultCcf();

        for (CustomerContext customer : ctx.getCustomers()) {
            // Step 1: 算表内 EAD（逐笔借据）
            for (AssetInput asset : customer.getAssets()) {
                double onBsEad = calcOnBsEad(asset);
                asset.setOnBsEad(onBsEad);
            }

            // Step 2: 算表外 EAD（按 product_type 分组）
            List<OffBsSubItem> offBsItems = calcOffBsEad(customer);

            // Step 3: 表外 EAD 分配到借据
            allocateOffBsToAssets(offBsItems, customer.getAssets());

            // Step 4: 汇总每条借据的 total_ead
            for (AssetInput asset : customer.getAssets()) {
                double total = asset.getOnBsEad() + asset.getOffBsEadAllocated();
                asset.setTotalEad(total);
            }
        }
    }

    /** 表内 EAD */
    private double calcOnBsEad(AssetInput asset) {
        if (asset.hasRepaymentSchedule()) {
            // 有还款计划：折现未来现金流 + 逾期欠款 + 应计利息
            double pv = 0;
            for (RepaymentSchedule installment : asset.getSchedule()) {
                double t = ChronoUnit.DAYS.between(asset.getCalcDate(), installment.getDueDate()) / 365.0;
                pv += (installment.getPrincipal() + installment.getInterest()) / Math.pow(1 + ctx.getDiscountRate(), t);
            }
            return pv + asset.getOverduePrincipal() + asset.getOverdueInterest() + asset.getAccruedInterest();
        } else {
            // 无还款计划：剩余本金 + 应计利息
            return asset.getOutstandingBalance() + asset.getAccruedInterest();
        }
    }

    /** 表外 EAD：按 product_type 分组，CCF 从高到低分配 */
    private List<OffBsSubItem> calcOffBsEad(CustomerContext customer) {
        List<OffBsSubItem> result = new ArrayList<>();

        // 按 product_type 分组
        Map<String, List<FacilityInfo>> byProduct = customer.getFacilities().stream()
            .filter(f -> f.getType() == FacilityType.OFF_BALANCE_SHEET)
            .collect(Collectors.groupingBy(FacilityInfo::getProductType));

        for (Map.Entry<String, List<FacilityInfo>> entry : byProduct.entrySet()) {
            List<FacilityInfo> items = entry.getValue();

            // 已开展的表外业务：直接用 (余额+利息) × CCF
            for (FacilityInfo item : items) {
                if (item.isDrawn()) {
                    double ccf = lookupCcf(item);
                    double ead = (item.getOutstandingBalance() + item.getAccruedInterest()) * ccf;
                    result.add(new OffBsSubItem(item, ead));
                }
            }

            // 已授信未开展：计算总剩余额度，按 CCF 从高到低分配
            double totalUndrawn = items.stream()
                .filter(f -> !f.isDrawn())
                .mapToDouble(f -> f.getTotalLimit() - f.getOutstandingBalance())
                .sum();

            List<FacilityInfo> undrawn = items.stream()
                .filter(f -> !f.isDrawn())
                .sorted(Comparator.comparingDouble(f -> -lookupCcf(f)))  // CCF 从高到低
                .toList();

            for (FacilityInfo item : undrawn) {
                double ccf = lookupCcf(item);
                double available = item.getTotalLimit() - item.getOutstandingBalance();
                double allocated = Math.min(available, totalUndrawn);
                double ead = allocated * ccf;
                result.add(new OffBsSubItem(item, ead));
                totalUndrawn -= allocated;
                if (totalUndrawn <= 0) break;
            }
        }
        return result;
    }

    /** 表外 EAD 分配到借据 */
    private void allocateOffBsToAssets(List<OffBsSubItem> offBsItems, List<AssetInput> assets) {
        for (OffBsSubItem item : offBsItems) {
            // 找到业务类型匹配的候选借据
            List<AssetInput> candidates = assets.stream()
                .filter(a -> a.getBusinessType().equals(item.getBusinessType())
                          && a.getProductType().equals(item.getProductType()))
                .toList();

            if (candidates.isEmpty()) {
                item.setException("Y");  // 无匹配借据，标记异常
                continue;
            }

            if (candidates.size() == 1) {
                // 1 对 1：全额分配
                candidates.get(0).addOffBsEad(item.getEadValue());
            } else {
                // 1 对 N：按未使用额度比例分配
                double totalUndrawn = candidates.stream()
                    .mapToDouble(a -> a.getTotalLimit() - a.getOutstandingBalance())
                    .sum();
                for (AssetInput asset : candidates) {
                    double ratio = (asset.getTotalLimit() - asset.getOutstandingBalance()) / totalUndrawn;
                    asset.addOffBsEad(item.getEadValue() * ratio);
                }
            }
        }
    }

    private double lookupCcf(FacilityInfo item) {
        return ccfCache.getOrDefault(key(item.getProductType(), item.getCommitmentType()), defaultCcf);
    }
}
```

### 6.5 LGD 计算引擎

**输入**：押品池（含各押品类型 + 评估价值）+ 池内借据（6.4 输出的 total_ead）
**输出**：lgd_value（押品池级，池内所有借据共享）

**设计说明：**
- 以押品池为起点计算池级 LGD，再赋值给池内每笔借据
- 池内 EAD 直接复用 6.4 引擎输出的 `total_ead`，不重复计算 CCF
- 参数（`lgd_base_value`、`discount_rate`、`depreciation_rate`）一次性加载到内存

**核心公式：**

```
LGD_pool = (EAD_uncovered × LGD_uncovered + EAD_covered × LGD_covered) / EAD_total

其中：
  EAD_total       = Σ(池内各借据.total_ead)
  押品净值         = 评估价值 × (1 + 折旧率) × (1 - 折扣率)
  押品净值汇总     = Σ(各押品净值)
  EAD_covered     = min(押品净值汇总, EAD_total)
  EAD_uncovered   = EAD_total - EAD_covered
  LGD_uncovered   = tbl_lgd_curve.lgd_base_value（按 group + collateral_type + product_type）
  LGD_covered     = LGD_uncovered × (1 - discount_rate)（推算，不冗余存储）
```

**核心逻辑：**

```java
public class LgdEngine implements EclEngine {

    // 参数缓存（一次性加载）
    private Map<String, Double> lgdCurveMap;     // (groupId, collType, prodType) → lgd_base_value
    private Map<String, Double> discountMap;     // (collateralType) → discount_rate
    private Map<String, Double> depreciationMap; // (collateralType, year) → depreciation_rate
    private double defaultLgd;                   // 方案级 LGD 缺省值

    @Override
    public void execute(JobContext ctx) {
        String schemeId = ctx.getSchemeId();
        loadParameterCache(schemeId);
        this.defaultLgd = ctx.getDefaultLgd();

        for (CustomerContext customer : ctx.getCustomers()) {
            // 按押品池分组处理
            Map<String, List<CollateralInfo>> poolMap = groupByCollateralPool(customer.getCollaterals());

            for (Map.Entry<String, List<CollateralInfo>> entry : poolMap.entrySet()) {
                String poolId = entry.getKey();
                List<CollateralInfo> collaterals = entry.getValue();

                // 找到该押品池关联的借据
                List<AssetInput> poolAssets = findAssetsByPool(customer.getAssets(), poolId);

                // Step 1: 汇总池内 EAD（复用 6.4 结果）
                double eadTotal = poolAssets.stream()
                    .mapToDouble(AssetInput::getTotalEad)
                    .sum();

                // Step 2: 计算押品净值汇总
                double netWorth = 0;
                String groupId = null;
                for (CollateralInfo coll : collaterals) {
                    double deprRate = depreciationMap.getOrDefault(
                        key(coll.getCollateralType(), coll.getAgeYears()), 0.0);
                    double discRate = discountMap.getOrDefault(coll.getCollateralType(), 0.0);
                    netWorth += coll.getAppraisedValue()
                        * (1 + deprRate)
                        * (1 - discRate);
                    if (groupId == null) groupId = coll.getGroupId();
                }

                // Step 3: 计算 LGD 参数
                double eadCovered = Math.min(netWorth, eadTotal);
                double eadUncovered = eadTotal - eadCovered;

                // LGD_uncovered：查 lgd_base_value（取池内首要押品类型的值）
                String primCollType = collaterals.get(0).getCollateralType();
                String primProdType = collaterals.get(0).getProductType();
                double lgdUncovered = lgdCurveMap.getOrDefault(
                    key(groupId, primCollType, primProdType), defaultLgd);

                // LGD_covered：从 LGD_uncovered 推算
                double discRate = discountMap.getOrDefault(primCollType, 0.0);
                double lgdCovered = lgdUncovered * (1 - discRate);

                // Step 4: 计算池级 LGD
                double lgdPool = (eadUncovered * lgdUncovered + eadCovered * lgdCovered) / eadTotal;

                // Step 5: 赋值给池内每笔借据
                for (AssetInput asset : poolAssets) {
                    asset.setLgdValue(lgdPool);
                }
            }

            // 无押品的借据：退化为 LGD_uncovered 路径
            for (AssetInput asset : customer.getAssets()) {
                if (asset.getLgdValue() == null) {
                    String key = key(asset.getGroupId(), "NONE", asset.getProductType());
                    double lgd = lgdCurveMap.getOrDefault(key, defaultLgd);
                    asset.setLgdValue(lgd);
                    asset.setLgdException("WARN");
                }
            }
        }
    }
}
```

### 7.3 ECL 计算

**输入**：ead_total + lgd_value + pd_details（6.3 已按 stage_result 转换好的 PD 值）
**输出**：ecl_weighted + ecl_details（多情景 ECL JSON）

**设计说明：**
- 纯公式计算，无条件匹配
- PD 值已由 6.3 按 stage_result 完成转换（Stage 1=12M / Stage 2=永续 / Stage 3=100%），7.3 不重算
- 情景权重由 7.3 统一加权，6.3 只透传 weight

**核心公式：**

```
For each scenario i in pd_details:
  ECL_i = EAD × pd_value_i × LGD

ecl_weighted = Σ(ECL_i × weight_i)
```

**核心逻辑：**

```java
public class EclCalculator {

    @Override
    public void execute(JobContext ctx) {
        for (CustomerContext customer : ctx.getCustomers()) {
            for (AssetInput asset : customer.getAssets()) {
                double ead = asset.getTotalEad();
                double lgd = asset.getLgdValue();
                List<PdDetail> pdDetails = asset.getPdDetails();

                List<EclDetail> eclDetails = new ArrayList<>();
                double eclWeighted = 0;

                // Step 1: 按情景计算 ECL
                for (PdDetail pd : pdDetails) {
                    // pd_value 已是 6.3 按 stage 转换后的值
                    double ecl = ead * pd.getPdValue() * lgd;
                    eclDetails.add(new EclDetail(pd, ecl));
                    eclWeighted += ecl * pd.getWeight();
                }

                asset.setEclDetails(eclDetails);
                asset.setEclWeighted(eclWeighted);
            }
        }
    }
}
```

### 7.4 管理层叠加

**输入**：ecl_weighted + ead_total + overlay 规则（按 group_id 过滤）+ scheme_id
**输出**：ecl_overlay_total + ecl_final + selected_overlay_id + effective_ratio

**设计说明：**
- 规则一次性加载到内存，用 ConditionEvaluator 解析 conditions JSON（与 6.2 一致）
- 多条规则命中时，只选等效调整比例最高的 1 条（非累加）
- 叠加基数固定为 EAD，作用于 ECL_FINAL

**核心逻辑：**

```java
public class OverlayEngine implements EclEngine {

    private List<OverlayRule> rules;  // 一次性加载

    @Override
    public void execute(JobContext ctx) {
        String schemeId = ctx.getSchemeId();
        this.rules = loadRules(schemeId);

        for (CustomerContext customer : ctx.getCustomers()) {
            for (AssetInput asset : customer.getAssets()) {
                processAsset(asset);
            }
        }
    }

    private void processAsset(AssetInput asset) {
        // Step 1: 按 group_id 过滤 + conditions 匹配
        List<OverlayRule> matched = new ArrayList<>();
        for (OverlayRule rule : rules) {
            if (!rule.getGroupId().equals(asset.getGroupId())) continue;
            if (ConditionEvaluator.evaluate(rule.getConditions(), asset.toFieldMap())) {
                matched.add(rule);
            }
        }

        if (matched.isEmpty()) {
            asset.setEclOverlayTotal(0);
            asset.setEclFinal(asset.getEclWeighted());
            return;
        }

        // Step 2: 计算各规则的等效比例，选最高
        OverlayRule selected = null;
        double maxRatio = -1;

        for (OverlayRule rule : matched) {
            double ratio = switch (rule.getAdjustmentType()) {
                case ADDBP -> rule.getAdjustmentValue() / 10000.0;
                case PERCENTAGE -> rule.getAdjustmentValue();
                case FIXED -> rule.getAdjustmentValue() / asset.getTotalEad();
            };
            if (ratio > maxRatio) {
                maxRatio = ratio;
                selected = rule;
            }
        }

        // Step 3: 算叠加金额
        double overlay = switch (selected.getAdjustmentType()) {
            case ADDBP -> asset.getTotalEad() * (selected.getAdjustmentValue() / 10000.0);
            case PERCENTAGE -> asset.getTotalEad() * selected.getAdjustmentValue();
            case FIXED -> selected.getAdjustmentValue();
        };

        // Step 4: 输出
        asset.setEclOverlayTotal(overlay);
        asset.setEclFinal(asset.getEclWeighted() + overlay);
        asset.setSelectedOverlayId(selected.getRuleId());
        asset.setEffectiveRatio(maxRatio);
    }
}
```

```java
public class FinalOutputLayer {
```

## 11. 非功能设计

### 10.1 性能

| 指标 | 目标 | 措施 |
|------|------|------|
| 批量计算 | 100万笔 ≤ 2 小时 | 并行引擎 + 分批提交 |
| API 响应 | P99 ≤ 3s | Redis 缓存 + 异步任务 |
| 并发 | 5 任务同时 | 线程池隔离 + 防重锁 |

**线程池**：core=8, max=16, 每批 5000 条, 每 10000 条提交

### 10.2 缓存

- 参数缓存：Redis，按 `scheme_id` 分区缓存，TTL=1 小时
- ECL 方案发布时清除对应 `scheme_id` 的缓存
- 试算查询：不写缓存（避免污染正式缓存）

### 10.3 异常处理体系

| 错误码 | 说明 | 级别 |
|--------|------|------|
| ECL_001 | PD查询无结果 | BLOCKING（阻断） |
| ECL_002 | EAD为负数 | WARN（强制为0） |
| ECL_003 | 无押品 | WARN（退化为无押品路径） |
| ECL_004 | 无生效方案 | BLOCKING |
| ECL_005 | 对账校验失败 | BLOCKING |
| ECL_006 | 方案状态不允许操作 | BLOCKING（如修改已生效方案） |

### 10.4 扩展性

- **情景可扩展**：tbl_pd_scenario 表支持动态添加情景
- **引擎 SPI**：新引擎实现 `EclEngine` 接口即可接入调度器
- **参数模块**：新参数类型只需添加以 `scheme_id` 为 FK 的表，天然支持版本控制
- **ECL 方案复制**：新模块加入时，只需在复制脚本中添加对应表的复制逻辑

### 10.5 开发规范

## 附录：关键业务规则速查

| 规则 | 逻辑 |
|------|------|
| 空值通配 | 明细字段 NULL → 通用匹配 |
| 优先级命中 | 多条命中取 priority 最小 |
| 兜底分组 | 无匹配 → GRP_DEFAULT |
| 6 维度全 NULL 禁止 | 禁止保存全通配明细 |
| 业务条线一致性 | 非零售×个人 → 非法 |
| PD 无兜底 | 查不到 pd_value → 阻断 |
| LGD 无兜底 | 查不到 lgd_base_value → 先查 NONE 路径，仍未找到 → 使用方案 default_lgd |
| CCF 无兜底 | 查不到 ccf_value → 使用方案 default_ccf（默认 0=不增加表外 EAD）|
| EAD 负数 | 强制为 0 |
| 无押品 LGD | 退化为 collateral_type='NONE' 的 lgd_base_value 路径 |
| EAD 折现率 | 有还款计划时按方案 discount_rate 折现未来现金流 |
| 方案缺省参数 | discount_rate / default_ccf / default_lgd 在方案总览页配置，跟随方案版本 |
| 叠加非累加 | 多条规则 → 取等效比例最高 1 条，非累加 |
| 叠加基数 EAD | 叠加金额 = EAD × 调整比例 |
| 叠加目标 ECL_FINAL | 叠加金额直接加到最终 ECL 上，不修改中间参数 |
| 对账校验 | abs(6.8 - 6.7) ≥ 0.01 → 阻断 |
| **DRAFT 新建/复制** | 新建或从生效方案复制后，方案状态 = DRAFT，可自由修改 |
| **EFFECTIVE 冻结** | EFFECTIVE 方案下所有参数只读，不可修改 |
| **阶段判断顺序** | 先 FORWARD 向前判定（按 priority）→ 再比较 last_stage → 改善才查 ROLLBACK |
| **回跳限制** | Stage 3→1 禁止；Stage 3→2 需观察期+还本付息；Stage 2→1 需评级回升+逾期消除 |
| **兜底阶段** | FORWARD 无匹配 → Stage 1，exception_flag = 'Y' |
| **回跳失败** | 不满足 ROLLBACK 规则 → 保持原阶段，exception_flag = 'Y' |
| **立即生效** | 发布时直接当场将方案切换为 EFFECTIVE |
| **计划生效** | 发布时设置 effective_date，到时间由批处理自动生效 |
| **生效批处理** | 计划生效：PUBLISHED + effective_date <= TODAY → EFFECTIVE；原方案 → EXPIRED |
| **正式跑批取生效方案** | 自动获取唯一 EFFECTIVE 方案 |
| **试算指定方案** | 支持任意指定方案 ID 重算 |
